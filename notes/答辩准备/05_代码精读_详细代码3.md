## 一、GamePanel 是什么(一句话职责)

> 📍 文件:[`game/GamePanel.java`](../../src/main/java/com/game/game/GamePanel.java) · 类定义 [L29](../../src/main/java/com/game/game/GamePanel.java#L29)

`GamePanel` 是游戏的「**屏幕 + 键鼠 + 时钟**」:

- **屏幕**:每帧把画面画出来(`paintComponent`)。
- **键鼠**:收键盘(WASD)和鼠标(瞄准/射击)事件(详见 [键盘与鼠标输入处理](../键盘与鼠标输入处理.md))。
- **时钟**:`javax.swing.Timer` 每 16ms 触发一帧(`tick`)。

和 `GameController` 的分工:**Panel 管"看得见的"**(渲染 / 输入 / Timer),**Controller 管"想得清的"**(逻辑)。Panel 不做任何游戏逻辑,只管"收集输入 → 喂给 Controller → Controller 算完 → Panel 画出来"。

## 四、`tick()` —— 每帧的入口 ⭐

> 📍 [`tick()` 第 138–145 行](../../src/main/java/com/game/game/GamePanel.java#L138)

```java
private void tick() {
    controller.setMoveDir(moveDirX(), moveDirY());   // ① 把按键状态喂给 Controller
    controller.update();                              // ② Controller 推进一帧逻辑
    repaint();                                        // ③ 请求重绘(→ paintComponent)
    if (!controller.isRunning()) timer.stop();        // ④ 游戏结束 → 停 Timer
}
```

**一帧 = ① 喂输入 → ② 算逻辑 → ③ 画画面 → ④ 结束则停。** Timer 每 16ms 自动调一次 tick,这就是游戏循环。

> `repaint()` 只是「**请求**重绘」——它排队,系统 EDT 空闲时才真正调 `paintComponent`。

## 六、`paintComponent()` —— 分层渲染 ⭐⭐⭐(核心)

> 📍 [`paintComponent(Graphics g)` 第 216–268 行](../../src/main/java/com/game/game/GamePanel.java#L216)

这是整个项目**最复杂的方法**,每帧画一次。核心是**分层绘制**(后画的盖在前面):

```java
@Override
protected void paintComponent(Graphics g) {
    super.paintComponent(g);                         // 清屏(父类)
    Graphics2D g2 = (Graphics2D) g;
    g2.setRenderingHint(KEY_ANTIALIASING, VALUE_ANTIALIAS_ON);  // 抗锯齿(圆/线条更平滑)
    int w = getWidth(); int h = getHeight();         // 画布实时尺寸

    // ─────── 固定层:底色(不随震屏偏移,防边缘露白)───────
    g2.setColor(UIStyle.BG); g2.fillRect(0, 0, w, h);

    // ─────── 世界层:受震屏整体偏移 ───────
    int sx = controller.getShakeX(); int sy = controller.getShakeY();
    g2.translate(sx, sy);                             // ⭐ 整个世界层偏移
    drawGrid(g2, w, h);                               // 网格地面
    controller.getPlayer().draw(g2);                  // 玩家
    for (Zombie z : controller.getZombies()) z.draw(g2);       // 僵尸
    for (Bullet b : controller.getBullets()) b.draw(g2);       // 子弹
    for (Particle p : controller.getParticles()) p.draw(g2);   // 粒子
    for (FloatingText ft : controller.getFloatingTexts()) ft.draw(g2); // 飘字
    g2.translate(-sx, -sy);                           // ⭐ 还原偏移!

    // ─────── 固定层:暗角 + HUD + 设置按钮(不随震屏)───────
    drawVignette(g2, w, h);                           // 暗角
    drawHud(g2);                                      // 分数/血条/难度
    drawExitButton(g2);                               // 右上角"设置"

    // ─────── 受伤红屏(全屏覆盖)───────
    float dmg = controller.getDamageFlash();
    if (dmg > 0f) { g2.setColor(new Color(220,30,30,(int)(dmg*110))); g2.fillRect(0,0,w,h); }

    // ─────── 准星 ───────
    drawCrosshair(g2);
}
```

### 三个关键设计 ⭐

**① 分「固定层」和「世界层」(震屏偏移)**

```
固定层(不 translate):背景色、暗角、HUD、设置按钮     ← 画面边框,不该跟着抖
世界层(translate(sx,sy)):网格、玩家、僵尸、子弹、粒子 ← 受伤时整体抖动
```

受伤时 Controller 设了 `shakeOffset`,世界层 `g2.translate(sx, sy)` 整体平移一点 → 画面"抖"。**但背景/HUD 不 translate** → 不跟着抖(否则边缘会露出底层颜色 = "露白")。

**② `translate` 后必须 `translate` 回来 ⭐**

```java
g2.translate(sx, sy);     // 偏移
... 画世界 ...
g2.translate(-sx, -sy);   // ⭐ 必须还原!
```

`Graphics2D` 的 translate 是**全局状态**——偏移后,后续**所有绘制**都会偏移,直到你 translate 回来。如果忘了还原(-sx,-sy),后面画的 HUD / 准星也会偏(甚至下次 paintComponent 的 super 调用也偏)。和 `AlphaComposite`(红闪 SRC_ATOP)一样:**改了全局状态必须还原**。

**③ 抗锯齿**

```java
g2.setRenderingHint(KEY_ANTIALIASING, VALUE_ANTIALIAS_ON);
```

打开后画的圆/线条边缘更平滑(不会锯齿)。代价是稍微多耗 CPU,但 2D 游戏这点开销忽略不计。

## 八、`drawExitButton()` + `isInExitButton()` —— 手动画的按钮

> 📍 [`drawExitButton` 第 381 行](../../src/main/java/com/game/game/GamePanel.java#L381) · [`isInExitButton` 第 371 行](../../src/main/java/com/game/game/GamePanel.java#L371) · [`exitX` 第 366 行](../../src/main/java/com/game/game/GamePanel.java#L366)

这个「设置」按钮**不是 JButton**,是画布上画的**一个红色圆角矩形 + 文字**,然后手动 hit-test:

```java
private void drawExitButton(Graphics2D g2) {
    int x = exitX();                    // = getWidth() - 84 - 10,贴右侧
    g2.setColor(new Color(220, 80, 80));
    g2.fillRoundRect(x, 10, 84, 28, 8, 8);   // 红色圆角矩形
    g2.drawString("设置", x + offset, 10 + offset);   // 白字居中
}

private boolean isInExitButton(int mx, int my) {
    return mx >= exitX() && mx <= exitX() + 84
        && my >= 10 && my <= 10 + 28;    // 矩形 hit-test
}
```

在 `mousePressed`(L98)里先调 `isInExitButton` 判断:点中红色矩形 = 开设置菜单;否则 = 射击。

> **为什么不用真 JButton**:游戏画布上放 Swing 组件(JButton)会和自定义渲染冲突(组件自己管理绘制/焦点/布局)。手动画 + hit-test = 完全自己控制,简单且一致。

## 十二、`start()` / `stop()` —— Timer 控制

> 📍 [`start()` 第 148 行](../../src/main/java/com/game/game/GamePanel.java#L148) · [`stop()` 第 155 行](../../src/main/java/com/game/game/GamePanel.java#L155)

```java
public void start() {
    timer.start();    // 启动 Timer → 开始 tick 循环
    SwingUtilities.invokeLater(this::requestFocusInWindow);  // 请求键盘焦点
}
public void stop() {
    timer.stop();     // 停 Timer → 游戏暂停(tick 不再被调)
}
```

- **`start`**:`timer.start()` 开始循环;`requestFocusInWindow` 让画布拿到键盘焦点(否则 WASD 不响应)。用 `invokeLater` 推迟到窗口焦点事务完成后再请求(某些平台首帧焦点抢不到)。
- **`stop`**:暂停 = 停 Timer。tick 不再被调 → update 不跑、frameCounter 不增、画面冻结 → **暂停时一切停**(包括时间)。

## 十三、GamePanel 全局一张图(怎么串起来)

```
Timer(16ms) → tick()
                │
    ┌───────────┼───────────┐
    ▼           ▼           ▼
 moveDirX/Y  controller    repaint()
 (读键盘)    .update()    → paintComponent()
             .setMoveDir    分层画:
    └──喂给──▶ (6步逻辑)    固定层(背景/HUD/暗角)
                            世界层(translate 震屏 → 网格/玩家/僵尸/子弹/粒子/飘字)
                            红屏 → 准星
```

**GamePanel 就是这条循环的「壳」**:Timer 驱动它转,输入通过它进 controller,画面通过它画出来。它自己**不做任何游戏逻辑**——全转发给 controller。

## 十五、小自测

1. **GamePanel 自己做不做游戏逻辑(刷怪/碰撞/计分)?**
2. **`paintComponent` 里为什么分「固定层」和「世界层」?**
3. **`g2.translate(sx,sy)` 之后为什么必须 `g2.translate(-sx,-sy)` 还原?**
4. **HUD(分数/血条)是用 JLabel 还是用 `drawString` 画?为什么?**
5. **「设置」按钮是 JButton 吗?点击怎么判断?**
6. **`repaint()` 和 `paintComponent()` 是什么关系?**
7. **暂停时 `timer.stop()` 后,游戏时间为什么也停了?**

<details><summary>答案</summary>

1. **不做**。GamePanel 只管渲染 + 输入 + Timer;逻辑全在 GameController。Panel 只是「转发器」。
2. 固定层(背景/HUD)不随震屏偏移,世界层(对象)偏移。如果背景也偏移,画面边缘会露出底层颜色(露白)。分两层只让世界层抖。
3. translate 改的是 Graphics2D 的**全局状态**,偏移后后面所有绘制都会偏,直到 translate 回来。不还原 = 后续画的全偏(甚至下一帧也偏)。
4. 用 **`drawString` / `fillRect` 画**。因为 HUD 每帧变(分数/血量),直接画和游戏画面一体;JLabel 会引入组件管理开销,不适合高频更新的游戏 HUD。
5. **不是 JButton**,是 `fillRoundRect` 画的一个红色矩形 + `drawString` 文字。点击用 `isInExitButton(x,y)` **手动矩形 hit-test** 判断(检查鼠标坐标是否落在矩形范围内)。
6. `repaint()` 是「请求重绘」(排队);`paintComponent()` 是「真正画」。调 repaint 不会立刻画,系统 EDT 空闲时才调 paintComponent。
7. 因为时间靠 `GameController.frameCounter / 60` 算;暂停 = `timer.stop()` → `tick` 不被调 → `update` 不跑 → `frameCounter` 不增 → 秒数冻结。

</details>

## 十六、`FloatingText` —— 浮动得分文字(+10 / +25)

> 📍 文件:[`game/FloatingText.java`](../../src/main/java/com/game/game/FloatingText.java)(60 行)
> 谁创建/清理:`GameController.handleCollisions` 打死僵尸时 `floatingTexts.add(new FloatingText(zx, zy, "+10", 黄, 40))`;每帧 `update` + `draw`,寿命尽了 `removeIf(FloatingText::isDead)` 清掉。**和 Particle 一样是用完即弃的纯视觉特效。**

### 字段一览

```java
private double x, y;              // 位置(y 会每帧减小 = 向上飘)
private final String text;        // 显示的文字("+10" / "+25")
private final Color color;        // 颜色(击杀用偏黄)
private int life;                 // 剩余寿命(帧)
private final int maxLife;        // 初始寿命(draw 算淡出比例用,final)
```

> 和 [`Particle`](../../src/main/java/com/game/game/Particle.java) **几乎同款结构**——都有 `life` / `maxLife` / `isDead()`,都用 `life/maxLife` 算 alpha 渐淡。区别:Particle 飞溅(有 vx/vy + 阻尼),FloatingText 只是**匀速向上飘**(没速度向量,固定 `y -= 0.8`)。

### `update()` —— 向上飘 + 寿命递减

```java
public void update() {
    y -= 0.8;      // 每帧上移 0.8 像素(固定速度,无阻尼)
    life--;        // 寿命 -1
}
```

就两行。比 Particle 更简单——**没有速度向量、没有阻尼**,只是匀速往上飘。为什么往上?因为得分提示往「天上」飘是游戏惯例(天上 = 好 = 正面反馈)。

### `draw()` —— 按寿命比例淡出

```java
public void draw(Graphics2D g) {
    float ratio = Math.max(0f, life / (float) maxLife);   // 剩余比例 1.0→0
    int alpha = (int) (ratio * 255);                       // 透明度 = 比例×255
    if (alpha <= 0) return;                                // 完全透明就不画
    g.setFont(new Font("微软雅黑", Font.BOLD, 16));
    g.setColor(new Color(color.getRed(), color.getGreen(), color.getBlue(), alpha));  // 4 参数 Color
    g.drawString(text, (int) x, (int) y);                 // 画文字
}
```

**和 `Particle.draw()` 同款逻辑**:`ratio = life/maxLife` → alpha = `ratio×255` → **越接近寿命终了越透明**。唯一区别是 Particle 画**圆**(`fillOval`),FloatingText 画**文字**(`drawString`)。

> 不用 SRC_ATOP:和 Particle 一样,直接画在画布上(文字本身没有"透明背景"要避让),用 4 参数 Color 带 alpha 就够了。

### `isDead()` —— 寿命尽

```java
public boolean isDead() { return life <= 0; }
```

`life` 归零 → Controller `floatingTexts.removeIf(FloatingText::isDead)` 清掉。

### 和 Particle 的对照(两个纯视觉特效)

| | `Particle` | `FloatingText` |
|---|---|---|
| 用途 | 火花/血液粒子 | 得分提示"+10" |
| 运动 | `x+=vx; y+=vy; vx*=0.92`(有速度+阻尼) | `y -= 0.8`(匀速上飘) |
| draw 内容 | `fillOval`(圆) | `drawString`(文字) |
| 淡出 | `ratio=life/maxLife → alpha` ✅ | 同款 ✅ |
| 实现 GameObject | ❌(纯视觉) | ❌(纯视觉) |
| 字段 | x/y/vx/vy/color/size/life/maxLife | x/y/text/color/life/maxLife |

> 两个类**共享「寿命驱动渐淡 + 用完即弃 + 不实现 GameObject」的同一个设计套路**,只是一个画圆(粒子)、一个画字(分数)。如果要加第三种特效(如伤害数字"-20"),完全照搬这个结构。
