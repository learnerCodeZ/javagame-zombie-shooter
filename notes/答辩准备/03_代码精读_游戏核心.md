## 开篇:游戏层全貌 + 和登录的根本区别

登录切片是 **事件驱动**——你点按钮,代码才跑。游戏是 **时间驱动**——有个 Timer 每秒触发 60 次,每次推进一帧,画面才"动"起来。

五个角色:

```
GamePanel(画布) ──持有──▶ GameController(大脑)
   │  │                       │ 持有
   │  Timer(每帧 tick)        ▼
   │  键盘/鼠标输入        Player / Zombie×N / Bullet×N   ← 都实现
   │  paintComponent 画         GameObject 接口
   ▼
controller.update() ──推进一帧──▶ 对象们 update() + 碰撞 + 刷怪 + 计分
```

- [`GamePanel`](../../src/main/java/com/game/game/GamePanel.java) = 画布 + `Timer` + 输入(键盘/鼠标)。
- [`GameController`](../../src/main/java/com/game/game/GameController.java) = 大脑,管所有对象、主循环逻辑、碰撞、计分。
- [`GameObject`](../../src/main/java/com/game/game/GameObject.java) = 接口,三个对象都实现它(多态)。
- [`Player`](../../src/main/java/com/game/game/Player.java) / [`Zombie`](../../src/main/java/com/game/game/Zombie.java) / [`Bullet`](../../src/main/java/com/game/game/Bullet.java) = 三个游戏对象。
- [`Difficulty`](../../src/main/java/com/game/game/Difficulty.java) = 难度参数(简单/困难两套数值)。

# §1 `GameObject` 接口 —— 多态的根基 ⭐(全文 22 行逐行讲)

> 文件:[`game/GameObject.java`](../../src/main/java/com/game/game/GameObject.java) —— 整个文件就 22 行,全贴出来逐行讲:

```java
package com.game.game;                       // ① 包:游戏层

import java.awt.Graphics2D;                  // ② 引入绘图类(draw 参数要用)

/**
 * 游戏对象统一接口:所有可更新、可绘制的实体都实现它。
 * <p>约定坐标以画布像素为单位,原点在左上角,x 向右、y 向下。
 */                                          // ③ 类注释:用途 + 坐标约定
public interface GameObject {                // ④ interface(接口,不是 class)

    /** 每帧推进自身状态(位置、血量等)。 */
    void update();                           // ⑤ 抽象方法 1:每帧推进

    /**
     * 在画布上绘制自身。
     * @param g 画布绘图上下文
     */
    void draw(Graphics2D g);                 // ⑥ 抽象方法 2:画自己
}
```

### 逐行拆解

**① `package com.game.game;`** —— 包声明。`game` 包放游戏层(对象/控制器/面板),和 `ui`/`dao`/`model`/`util` 平级。对应目录 `src/main/java/com/game/game/`。

**② `import java.awt.Graphics2D;`** —— 引入 `Graphics2D`(Java 2D 绘图类)。`draw(Graphics2D g)` 的参数类型,用它画圆/线/贴图。

**③ 类注释里的「坐标约定」** —— **重点**:"原点在左上角,x 向右、y 向下"。这是**计算机图形学的惯例**(和数学坐标 y 朝上相反)。项目里所有坐标(玩家 x/y、僵尸位置、子弹、碰撞)都遵循它。记住 **y 向下**——所以"僵尸在玩家上方"= 僵尸的 y 比玩家**小**。

**④ `public interface GameObject`** —— `interface`(接口),不是 `class`。接口**只规定"有哪些方法",不写实现**(方法体空的,连 `{}` 都没有,直接 `;`)。谁要用就 `implements GameObject` 自己写实现。

**⑤ `void update()`** —— 抽象方法 1:**每帧推进自身状态**。Player 的 update 衰减红闪、Zombie 的 update 追踪玩家移动、Bullet 的 update 沿速度向量飞——每个实现类自己决定"推进"干啥。接口只规定"你必须有个 update"。

**⑥ `void draw(Graphics2D g)`** —— 抽象方法 2:**在画布上画自己**。参数 `g` 是画布的"画笔"(`Graphics2D`),实现类用 `g.fillOval(...)`/`g.drawImage(...)` 等画。接口不管你怎么画,只规定"你必须能被画"。

### 接口 vs 抽象类(对比)

两者都能"规定子类必须实现某些方法",但有本质区别。用最简单的例子看:

**接口(`interface`)** —— 只定"能做什么",不含字段 / 构造器 / 实现:

```java
public interface Flyable {
    void fly();                    // 只有抽象方法,没有实现体
}
class Bird  implements Flyable { public void fly() { System.out.println("鸟扇翅膀"); } }
class Plane implements Flyable { public void fly() { System.out.println("飞机喷气"); } }
// 鸟和飞机没有"是谁"的关系,但都能"飞"——用接口
```

**抽象类(`abstract class`)** —— 带"是什么",可含字段 + 构造器 + 已实现的方法(子类共享):

```java
public abstract class Animal {
    String name;                   // ✅ 可以有字段
    Animal(String n) { name = n; } // ✅ 可以有构造器
    void breathe() { /* 呼吸 */ }  // ✅ 可以有已实现的方法(子类直接继承,共享代码)
    abstract void sound();         // 抽象方法,子类必须实现
}
class Dog extends Animal {
    Dog(String n) { super(n); }
    void sound() { System.out.println("汪汪"); }
}
// Dog 是一种 Animal(is-a),共享 breathe 和 name 字段——用抽象类
```

**一张表收尾**:

| | 接口 `interface` | 抽象类 `abstract class` |
|---|---|---|
| 用法 | `implements` | `extends` |
| 能几个 | 一个类可实现**多个**接口 | 一个类只能继承**一个**(单继承) |
| 字段 | 只能 `public static final` 常量 | 任意字段 |
| 构造器 | ❌ 没有 | ✅ 有 |
| 已实现的方法 | Java 8 起才有 `default` | ✅ 可以有,子类共享 |
| 语义 | **"能做什么"**(can-do / 行为契约) | **"是什么"**(is-a / 模板) |

**回到本项目:为什么 `GameObject` 用接口?**

`Player`(玩家)/`Zombie`(敌人)/`Bullet`(子弹)是三类**完全不同的东西**——它们**没有 is-a 关系**(玩家不是僵尸、子弹不是玩家),也**没有共享的实现代码**(各自的 update/draw 写法完全不同)。唯一的共同点是"都需要每帧 update + 被 draw"——这是个**行为约定**,不是身份关系。所以接口正合适。

> 反过来:如果三个对象**有共享代码**(比如都有 x/y 坐标 + 都要钳制在画布内),那把共享部分放进抽象类、子类继承会更省事(避免重复)。本项目把位置管理挪到了 `GameController`(§2.1 讲过 Player.update 不自己移动),三个对象不需要共享代码,所以接口就够、抽象类反而多余。

### 为什么用接口 —— 多态的根基 ⭐

三个对象(`Player`/`Zombie`/`Bullet`)都 `implements GameObject`,于是 `GameController` 能用**统一循环**调它们:

```java
for (Zombie z : zombies) { z.update(); }   // 普通僵/壮汉都当 Zombie 调,各跑各的 update
for (Bullet b : bullets) { b.update(); }
```

**多态**:同一句 `z.update()`,NORMAL 和 BRUTE 僵尸各跑各的实现;同一句 `update()`,Player/Zombie/Bullet 表现完全不同。**调用方不用管具体类型,只管"它是个能 update、能 draw 的 GameObject"**。

**开闭原则**:以后加「道具」「陷阱」新对象,只要 `implements GameObject`,面板的遍历代码**一行都不用改**就能接纳它——对扩展开放、对修改关闭。

> 答辩金句:「我用 `GameObject` 接口抽象出'能 update、能 draw'的游戏对象,`Player/Zombie/Bullet` 都实现它,体现多态;以后加新对象不用改遍历逻辑,符合开闭原则。」

## 2.1 `Player` —— 玩家

> 文件:[`game/Player.java`](../../src/main/java/com/game/game/Player.java)
> 📍 关键行:L22 `SPEED=4.0` · L29 `hp=100` · L31 `radius=26` · **L46-53 `update()` 只衰减红闪** · L71-123 `draw()`(精灵图+红闪/回退画橙猫) · L130-136 `takeDamage()`

**字段一览(逐个注释)**:

```java
static final double SPEED = 4.0;        // 移动速度(像素/帧);控制器和画布共用,统一来源
private static BufferedImage SPRITE;    // 玩家精灵图(透明 PNG),类级懒加载;读失败→null→draw 回退画橙猫
private double x;                        // 横坐标(画布像素,原点在左上)
private double y;                        // 纵坐标(y 向下)
private int hp = 100;                    // 血量;被僵尸撞扣,归零则本局结束
private double angle;                    // 枪口朝向角度(弧度);由鼠标位置 atan2(dy,dx) 算出
private final int radius = 26;           // 圆形半径(像素);碰撞判定 + 决定贴图大小
private int hitFlash;                    // 受伤红闪剩余帧数(>0 时 draw 给角色染红)
```

> **`BufferedImage` 是什么**:`java.awt.image.BufferedImage` 是 Java 的「**存在内存里的图片**」——把一张图的像素数据装进内存缓冲区(buffer),可以快速反复贴到画布上。游戏里用它存角色精灵图(`player.png`):首次用时从磁盘读一次进 `BufferedImage`,之后每帧 `g.drawImage(SPRITE, ...)` 直接从内存取来贴(极快,不用每帧重读磁盘)。读图失败(`ImageIO.read` 抛异常)就保留 `null`,`draw` 改用代码画橙色小猫兜底,保证不崩。这里 `SPRITE` 是 **`static`** = 所有 Player 实例共用同一张图(省内存)、**懒加载** = 第一次画时才读(不在启动时就加载)。Zombie 类里也有个同款的 `SPRITE`,原理一样。

**`player.png` 是怎么进 `BufferedImage`(内存)的?** —— 懒加载 + 缓存

`SPRITE` 初始是 `null`,真正读图在 `sprite()` 方法里(`draw` 每帧会调它):

```java
private static BufferedImage SPRITE;          // ① 类级静态字段,初始 null(= "缓存"槽位)

private static BufferedImage sprite() {
    if (SPRITE == null) {                      // ② 还没加载过?只第一次进
        try {
            SPRITE = ImageIO.read(             // ③ ImageIO 把 PNG 字节解码成 BufferedImage(像素进内存)
                Player.class.getResourceAsStream("/images/player.png"));  // ④ 从 classpath 读文件
        } catch (IOException | IllegalArgumentException e) {
            SPRITE = null;                      // ⑤ 读失败:保持 null,draw 回退画橙猫
        }
    }
    return SPRITE;                             // ⑥ 已加载:直接返回内存里的图(不再碰磁盘)
}
```

每帧 `draw()` 里 `BufferedImage sp = sprite();`,流程:

1. **第一帧**:`SPRITE` 是 `null` → 进 `if` → `ImageIO.read` 把 `player.png` 从磁盘**解码成像素数据**、存进 `SPRITE`(进内存/缓存)。
2. **之后每帧**:`SPRITE` 已非 `null` → 跳过 `if` → 直接返回 `SPRITE`(内存取数,**不读磁盘**)。

逐点拆:

- **`ImageIO.read(输入流)`**:Java 的图片加载 API,把图片字节流**解码**成一张 `BufferedImage`(像素数据放内存)。PNG 的压缩字节 → 解码 → 内存里的像素矩阵。
- **`getResourceAsStream("/images/player.png")`**:从 **classpath** 读文件。`src/main/resources/images/player.png` 编译后打进 classpath,所以用 `/images/player.png` 读;返回 `InputStream`(原始字节流)。**不依赖工作目录**,换机器也能读到位。
- **`static SPRITE`**:**类级**字段,所有 `Player` 实例共用一份图,不每个对象存一份(省内存)。
- **`if (SPRITE == null)`**:**懒加载**——第一次用到才读,不在程序启动时就加载(启动更快)。
- **try/catch**:文件缺失/损坏 → `SPRITE` 留 `null` → `draw` 回退画橙猫,保证不崩。

> 这就是经典的「**懒加载 + 缓存**」模式:**第一次**用时从磁盘读到内存(慢一次),**之后**反复用内存里的(快)。游戏每帧画几十上百次,若每帧都 `ImageIO.read` 读磁盘会卡死;缓存成 `BufferedImage` 后,每帧只是内存取数,极快。**回退兜底**(读失败画橙猫)保证"图读不到游戏也不崩"。

**一个反直觉点**:`Player.update()`(L46)**不移动自己**——它只衰减受伤红闪计时。位置是谁改的?是 `GameController.update()` 算好新位置后 `player.setX/Y()` **回填**的。

> 为什么这样设计?移动要**钳制在画布内 + 归一化斜走**(避免对角线超速),这逻辑放控制器统一做,Player 只管"我被放在哪"。职责分离:Player 管"我是谁(血量/角度/外观)",Controller 管"世界怎么动"。

- `draw()`:优先贴 `player.png` 精灵图(按瞄准方向左右翻转),读图失败**回退画橙色小猫**(耳朵+身体+眼睛+枪管),保证不崩。
- `takeDamage(dmg)`:扣血 + 触发红闪(`hitFlash=10`)。血量钳到 0。

## 2.2 `Zombie` —— 僵尸

> 文件:[`game/Zombie.java`](../../src/main/java/com/game/game/Zombie.java)
> 📍 关键行:L24 `Type{NORMAL, BRUTE}` 枚举 · L71-89 构造(按 type 定半径/血量/分值) · **L91-110 `update()` 追踪玩家** · L204-207 `takeDamage()` · **L212-214 `kill()`(撞玩家后消失,不计分)** · L238-240 `getScoreValue()`

**两种变体**,属性都跟 `Type` 走:

| Type | 半径 | 血量 | 分值 | 速度 | 速度来源 |
|---|---|---|---|---|---|
| NORMAL(普通绿僵) | 18 | 1 | 10 | 较快 | `1.0 + random` |
| BRUTE(壮汉) | 26 | 3 | 25 | 较慢 | `0.6 + random*0.4` |

- `update()`(L91):**追踪**——`setTarget(玩家位置)` 由控制器每帧回填,僵尸朝目标归一化移动(`dx/dist * speed`)。所以僵尸永远朝你追。
- `takeDamage(dmg)`(L204):扣血 + 闪白(`hitFlash=6`)。血量≤0 → `isDead()` 真 → 被控制器清理 + 计分。
- `kill()`(L212):**直接 hp=0,但不计分**。用于"僵尸撞到玩家后消失"——它撞了你得消失(否则一只僵尸能反复扣血),但它不是被你打死的,不给分。**两个"死"的语义**:`takeDamage` 死 = 击杀(计分);`kill` 死 = 消失(不计分)。

# §3 `Difficulty` 枚举 —— 参数化难度(依赖注入)

> 文件:[`game/Difficulty.java`](../../src/main/java/com/game/game/Difficulty.java)
> 📍 关键行:L22 `EASY(...)` · L25 `HARD(...)` · L28-38 字段(spawnInterval/minSpawn/bruteMinSec/bruteChance/hitDamage/label)

```java
public enum Difficulty {
    EASY(90, 30, 10, 0.18, 20, "简单"),
    HARD(60, 20,  5, 0.35, 25, "困难");
    public final int spawnInterval, minSpawn, bruteMinSec, hitDamage;
    public final double bruteChance;
    public final String label;
}
```

**为什么不用 `static final` 常量**?常量是**全局唯一**的(整个程序就一份),要做"简单一套、困难一套"两套参数,常量做不到。枚举字段 = **每个档位一套独立参数**,构造时注入 `GameController`(`new GameController(difficulty)`),运行时读 `difficulty.spawnInterval` 等。

> 这是**依赖注入/参数化**思想:同一套代码,注入不同数值 → 不同难度体验。改难度只动这个枚举,`GameController` 里没有任何写死的魔法数。

# §4 `GameController` —— 游戏大脑 ⭐⭐

> 文件:[`game/GameController.java`](../../src/main/java/com/game/game/GameController.java)

## 4.1 `update()` 单帧推进 6 步 ⭐ 核心

> 📍 [`update()` 第 139–193 行](../../src/main/java/com/game/game/GameController.java#L139)

```java
public void update() {
    if (!running) return;
    // 0) 应用移动输入(WASD):斜走归一化 + 钳制在画布内 → player.setX/Y
    // 1) 推进所有对象状态(僵尸先 setTarget 玩家位置,再 update 移动;子弹 update)
    // 1.5) 推进特效(粒子/浮动文字/震屏/红屏)并清理已消亡特效
    // 2) 按节奏刷怪:间隔随存活秒数缩短
    int interval = Math.max(difficulty.minSpawn, difficulty.spawnInterval - getElapsedSec());
    if (frameCounter % interval == 0) spawnZombie();
    // 3) 碰撞检测(只改状态,不增删集合!)
    handleCollisions();
    // 4) 清理:出界/已死的子弹、已死的僵尸(用 removeIf)
    bullets.removeIf(b -> b.isDead() || b.isOffscreen(WIDTH, HEIGHT));
    zombies.removeIf(Zombie::isDead);
    // 5) 结束判定
    if (player.getHp() <= 0) { running = false; fireGameOver(); }
}
```

**⭐ 最关键的工程点:第 3 步「碰撞只改状态」和第 4 步「清理用 `removeIf`」分开做。**

为什么?因为 `for (Zombie z : zombies)` 遍历时**不能增删集合**——否则抛 `ConcurrentModificationException`。所以碰撞检测里只**标记**(`z.takeDamage`、`b.setDead(true)`),等遍历完,再用 `removeIf` 一次性清理被标记的。这是 Java 集合遍历删除的经典套路。

## 4.2 `shoot()` —— 开火

> 📍 [`shoot()` 第 123–133 行](../../src/main/java/com/game/game/GameController.java#L123)

```java
double muzzle = player.getRadius() + 8;          // 枪口离玩家中心
double bx = player.getX() + Math.cos(angle) * muzzle;
double by = player.getY() + Math.sin(angle) * muzzle;
double vx = Math.cos(angle) * BULLET_SPEED;       // 子弹速度向量
double vy = Math.sin(angle) * BULLET_SPEED;

```

`angle` 是鼠标方向(由 `setAim` 算 `atan2(dy,dx)`)。`cos/sin` 把角度拆成 x/y 两个分量——子弹沿瞄准方向飞。

## 4.3 `spawnZombie()` —— 刷怪(难度递增 + 公平)⭐

> 📍 [`spawnZombie()` 第 199–236 行](../../src/main/java/com/game/game/GameController.java#L199)

两个设计点:

**(a) 难度随时间递增**(L200):
```java
int interval = Math.max(difficulty.minSpawn, difficulty.spawnInterval - getElapsedSec());
```
存活越久 `getElapsedSec()` 越大 → `interval` 越小 → 刷怪越密。但有 `minSpawn` 封顶,不会失控。简单档:90 帧→30 帧;困难档:60 帧→20 帧。

**(b) Brute 门槛 + 概率**(L200):
```java
boolean brute = getElapsedSec() >= difficulty.bruteMinSec && random.nextDouble() < difficulty.bruteChance;
```
存活满 `bruteMinSec` 秒(简单 10s、困难 5s)后,按 `bruteChance`(简单 18%、困难 35%)概率刷壮汉。前期只有普通僵,后期混壮汉。

**(c) 生成安全距离**(L206-233):生成点离玩家必须 ≥ `safeDist`(玩家半径+僵尸半径+40),最多重试 10 次。否则贴墙玩家会被"凭空"生成在身上的僵尸第一帧撞到(不公平,见[审计修复②](../v1.0_审计修复/扩展_审计修复_数据与玩法.md))。

## 4.4 `handleCollisions()` —— 圆形碰撞 ⭐

> 📍 [`handleCollisions()` 第 241–285 行](../../src/main/java/com/game/game/GameController.java#L241)

```java
// ① 子弹 × 僵尸:圆心距 < 半径和 → 命中
double dx = b.getX() - z.getX();
double dy = b.getY() - z.getY();
if (Math.hypot(dx, dy) < b.getRadius() + z.getRadius()) {
    z.takeDamage(1);            // 僵尸扣血
    b.setDead(true);            // 子弹标记删除
    spawnHitSparks(...);        // 命中火花
    SoundUtil.hit();
    if (z.isDead()) {           // 打死了
        killCount++;
        score += z.getScoreValue();   // +10 / +25
        spawnZombieDeath(z);          // 血溅
        floatingTexts.add(new FloatingText(..., "+" + z.getScoreValue(), ...));
    }
    break;    // ⭐ 一颗子弹只命中一只僵尸,然后跳出内层循环
}
// ② 僵尸 × 玩家:撞到 → 玩家扣血 + 僵尸 kill() 消失(防反复扣)
if (Math.hypot(...) < z.getRadius() + player.getRadius()) {
    player.takeDamage(difficulty.hitDamage);
    SoundUtil.hurt();
    z.kill();                   // 消失但不计分
    triggerShake(14, 6);        // 震屏
    damageFlashFrames = ...;    // 红屏
}
```

**圆形碰撞**公式:两圆心距 `< 半径之和` → 相交。比矩形 AABB 更贴合圆形角色。
**`break`**:`break` 让一颗子弹命中一只僵尸后**立刻退出内层循环**,不再检测其他僵尸——所以一枪只能杀一只(不会穿透)。

## 4.5 特效 + 计时

- **特效**:粒子(枪口火花 `spawnMuzzleFlash`/命中火花/僵尸爆血 `spawnZombieDeath`)、震屏(`triggerShake` 线性衰减)、红屏(`damageFlashFrames`)、浮动文字(`FloatingText` "+10")。都由 `GamePanel` 在绘制时读取并渲染。
- **`getElapsedSec()`**(L465):基于 `frameCounter / 60` 折算秒数。**暂停不计入**(因为暂停时 `update()` 不被调、`frameCounter` 不增)——所以存档的存活时间、刷怪节奏、Brute 门槛都只反映**实际游玩时间**。

# §5 `GamePanel` —— 画布 + Timer + 输入 ⭐⭐

> 文件:[`game/GamePanel.java`](../../src/main/java/com/game/game/GamePanel.java)

## 5.1 `javax.swing.Timer` 主循环 ⭐

> 📍 L105 `new Timer(16, e -> tick())` · [`tick()` 第 111–118 行](../../src/main/java/com/game/game/GamePanel.java#L111)

```java
this.timer = new Timer(16, e -> tick());   // 16ms 一帧 ≈ 60 FPS

private void tick() {
    controller.setMoveDir(moveDirX(), moveDirY());   // 把按键状态喂给控制器
    controller.update();                              // 推进一帧逻辑
    repaint();                                        // 触发 paintComponent 重绘
    if (!controller.isRunning()) timer.stop();        // 游戏结束 → 停 Timer
}
```

- `javax.swing.Timer(间隔, 监听器)`:每隔 16ms 触发一次 `tick()`。**它的回调在 EDT 上**(01 讲过 EDT)。
- **一帧 = 逻辑推进(`update`) + 重绘(`repaint`)**。这就是游戏循环:不停"算一帧 + 画一帧"。
- `start()`/`stop()`:暂停 = `stop()`(Timer 不再 tick,游戏冻结);恢复 = `start()`。

## 5.2 输入:键盘 + 鼠标

> 📍 L58-72 鼠标移动 · L74-85 鼠标点击 · L87-97 键盘 · L99-104 失焦清按键

- **键盘**:`pressedKeys`(一个 `Set<Integer>` 键码)记录按住的键。`keyPressed` 加、`keyReleased` 减。`moveDirX/Y()`(L133-154)合成方向:右-左、下-上。**用集合而非单个变量**,所以能同时按住多键(如 W+D 斜走),松开一个不影响另一个。
- **鼠标移动** → `controller.setAim(x, y)`(算瞄准角度)。
- **鼠标左键** → `controller.shoot()`(先判是不是点「设置」按钮)。
- **失焦清按键**(L99):窗口失焦时 `pressedKeys.clear()`——否则"按住 W 切到别的窗口,回来角色还在一直走"。

## 5.3 `paintComponent` 分层绘制 ⭐

> 📍 [`paintComponent` 第 165–211 行](../../src/main/java/com/game/game/GamePanel.java#L165)

绘制顺序(后画的盖在前面):

```
1. 背景色(固定层,不随震屏动)         ← 避免震屏时边缘露白
2. 世界层(受震屏偏移 translate):
   网格 → Player → Zombies → Bullets → 粒子 → 浮动文字
3. 暗角 + HUD(分数/血条/难度) + 设置按钮(固定层)
4. 受伤红屏(全屏半透明红覆盖)
5. 准星(鼠标位置)
```

**关键点**:背景、HUD、暗角是**固定层**(不 `translate`),只有世界层随震屏 `translate(sx, sy)`。如果背景也跟着震,屏幕边缘就会露出底层颜色(露白)。

## 5.4 鼠标锁定(后加的)

> 📍 `grabMouseBack()`:游戏中鼠标移出画布 → 推回出界点内侧 8px;暂停/切窗放开。

详见 [`扩展_鼠标锁定与窗口固定`](../v1.0_审计修复/扩展_鼠标锁定与窗口固定.md)。两道开关:`timer.isRunning()`(只在游戏跑着时锁)+ `win.isActive()`(切到别的窗口不抢鼠标)。

# §6 串起来:一帧的完整生命周期

```
javax.swing.Timer 每 16ms 触发
        │ (在 EDT 上)
        ▼
GamePanel.tick()
        │
        ├─ controller.setMoveDir(按键合成方向)
        ├─ controller.update()        ← 大脑推进一帧
        │     ├─ 0) 移动玩家(钳制在画布内)
        │     ├─ 1) 推进僵尸/子弹 update()
        │     ├─ 1.5) 推进特效
        │     ├─ 2) 按节奏刷怪(难度递增)
        │     ├─ 3) 碰撞检测(只改状态)
        │     ├─ 4) removeIf 清理失效对象
        │     └─ 5) 血量≤0 → fireGameOver()
        ├─ repaint()                   ← 触发重绘
        │     └─ paintComponent 分层画(背景→世界→HUD→红屏→准星)
        └─ !running → timer.stop()
```

**死亡流程**:`update()` 检测 `hp<=0` → `fireGameOver()` → `onGameOver` 回调 → `GameWindow.settle()`(写战绩 + 弹结算面板,见 01 站 GameWindow)。

# §8 自测

1. **`Player.update()` 为什么不自己移动位置?**
2. **碰撞检测为什么"只改状态、清理用 `removeIf` 分开"?**
3. **难度怎么随时间递增的?有封顶吗?**
4. **一颗子弹能命中几只僵尸?为什么?**
5. **`javax.swing.Timer` 的回调在哪条线程?游戏逻辑跑在 EDT 有什么隐患?**
6. **`paintComponent` 里背景和 HUD 为什么不随震屏 `translate`?**

<details><summary>参考答案</summary>

1. 移动要**归一化斜走 + 钳制在画布内**,这逻辑放 `GameController` 统一做,算好后 `player.setX/Y` 回填。职责分离:Player 管"我是谁",Controller 管"世界怎么动"。
2. `for` 遍历集合时**不能增删**元素,否则抛 `ConcurrentModificationException`。所以碰撞里只**标记**(`takeDamage`/`setDead`),遍历完用 `removeIf` 统一清理被标记的。
3. `interval = max(minSpawn, spawnInterval - elapsedSec)`:存活越久 interval 越小(刷越密),`minSpawn` 是封顶,不会失控。
4. **一只**。`handleCollisions` 里命中后 `break` 退出内层循环,不再检测其他僵尸。
5. **EDT**(事件分派线程)。隐患:游戏逻辑跑在 EDT,如果某帧逻辑很重(或 DB 查询)会**卡住界面**(EDT 同时管绘制)。本项目逻辑轻还好;DAO 存档是已知 EDT 隐患([已知问题 #3](../../docs/已知问题与后续优化.md)),生产应用 `SwingWorker`。
6. 震屏是整体 `translate` 世界层;如果背景/HUD 也 translate,屏幕边缘会露出底层颜色(露白)。所以固定层不 translate。

</details>
