## 一、`Difficulty` —— 难度档位(枚举参数化)⭐

> 文件:[`game/Difficulty.java`](../../src/main/java/com/game/game/Difficulty.java)(49 行)
> 一句话职责:把"随难度而变"的刷怪/伤害参数,集中到 `EASY`/`HARD` 两个枚举常量里,构造 `GameController` 时注入。
> 为什么单拎出来讲:它体现了 **enum(枚举)+ 参数化/依赖注入** 两个重要 Java 思想,而且是**答辩亮点(难度系统)**。

### 1. 先理解 `enum`(枚举)是什么

`enum` 是 Java 一种特殊类型,用来表示**一组固定的命名常量**。比如季节只有春夏秋冬、难度只有简单困难——这种"可数、固定"的值,就用 enum。

最简单的 enum:

```java
public enum Season { SPRING, SUMMER, AUTUMN, WINTER }
```

`SPRING`/`SUMMER`/... 就是 4 个常量,类型是 `Season`。用的时候 `Season s = Season.SUMMER`。

> enum 比"用 int 常量表示"安全:`if(s == 5)` 容易写错(5 是啥?),`if(s == Season.SUMMER)` 一眼看懂,而且编译器会检查你传的是不是合法的 `Season` 值。

### 2. enum 还能带「字段 + 构造器」(本项目用的就是这种)⭐

`Difficulty` 不只是两个空名字,它**每个常量都自带一组参数**——这是 enum 的高级用法(把 enum 当"迷你类"):

```java
public enum Difficulty {
    EASY(90, 30, 10, 0.18, 20, "简单"),     // ← EASY 这个常量,带 6 个参数
    HARD(60, 20,  5, 0.35, 25, "困难");      // ← HARD 这个常量,带另一组 6 个参数

    public final int spawnInterval;          // 字段(每个常量各有一份)
    public final int minSpawn;
    public final int bruteMinSec;
    public final double bruteChance;
    public final int hitDamage;
    public final String label;

    Difficulty(int spawnInterval, int minSpawn, int bruteMinSec,        // 构造器(enum 内部用)
               double bruteChance, int hitDamage, String label) {
        this.spawnInterval = spawnInterval; ...
    }
}
```

理解:`EASY(90,30,10,0.18,20,"简单")` 就是在**调构造器** `Difficulty(90,30,10,0.18,20,"简单")`,把 EASY 这个常量的字段填成这组值。同理 HARD 填另一组。**所以 EASY 和 HARD 各自携带一套不同的难度参数**。

### 3. 六个参数逐个讲(对照 EASY / HARD)

| 字段 | 类型 | 含义 | EASY | HARD | 在哪用 |
|---|---|---|---|---|---|
| `spawnInterval` | int(帧) | 初始刷怪间隔 | 90 帧(1.5 秒) | 60 帧(1.0 秒) | `GameController.spawnZombie` 算间隔 |
| `minSpawn` | int(帧) | 间隔的封顶下限(越打越快的极限) | 30 帧(0.5 秒) | 20 帧(0.33 秒) | 同上 |
| `bruteMinSec` | int(秒) | 壮汉开始出现的存活门槛 | 10 秒 | 5 秒 | `spawnZombie` 判 Brute |
| `bruteChance` | double | 刷怪替换成壮汉的概率 | 0.18(18%) | 0.35(35%) | 同上 |
| `hitDamage` | int | 僵尸撞玩家的伤害 | 20 | 25 | `handleCollisions` 扣血 |
| `label` | String | 中文显示名 | "简单" | "困难" | HUD / 窗口标题 / 菜单弹窗 |

> **帧 ↔ 秒换算**:项目 60 帧 = 1 秒(`GameController.FRAMES_PER_SEC = 60`)。所以 `spawnInterval=90` 帧 = 1.5 秒刷一只。HARD 刷得更密(1.0 秒)、壮汉更早(5 秒 vs 10 秒)、概率更高(35% vs 18%)、撞一下更疼(25 vs 20)——**难度全靠这 6 个数拉开**。

### 4. 为什么用 enum,不用 `static final` 常量?⭐⭐(核心)

假设不用 enum,用普通常量:

```java
public class GameConfig {
    public static final int EASY_SPAWN = 90;
    public static final int HARD_SPAWN = 60;
    public static final int EASY_DAMAGE = 20;
    public static final int HARD_DAMAGE = 25;
    ... // 6 个参数 × 2 档 = 12 个常量
}
```

然后 `GameController` 里:

```java
if (difficulty == EASY) {
    interval = EASY_SPAWN; damage = EASY_DAMAGE; ...
} else {
    interval = HARD_SPAWN; damage = HARD_DAMAGE; ...
}
```

问题:**每加一个参数,就要加两个常量 + 改这个 if-else**;而且 `GameController` 里到处是 `if(easy)...else...`,又长又容易漏。

enum 的写法(本项目):

```java
private final Difficulty difficulty;     // GameController 里,一个字段搞定
...
int interval = Math.max(difficulty.minSpawn, difficulty.spawnInterval - getElapsedSec());
player.takeDamage(difficulty.hitDamage);
```

**一套代码,读 `difficulty.xxx` 就自动用对应档位的值**——不用 if-else、不用知道当前是简单还是困难。这就是**依赖注入 / 参数化**:把"随难度变的数值"抽出来做成 enum 字段,注入 `GameController`,代码就变成"通用逻辑 + 参数"。

> **答辩金句**:「我把难度参数封装进 `Difficulty` 枚举,`GameController` 构造时注入;代码只读 `difficulty.xxx`,同一套逻辑跑两档数值——这是参数化 / 依赖注入思想,加新难度(比如『地狱』)只要加一个枚举常量,代码不用改。」

### 5. 怎么被 `GameController` 用(注入 + 读字段)

```java
// GameController 构造(MainFrame 选难度时 new GameController(Difficulty.HARD))
public GameController(Difficulty difficulty) {
    this.difficulty = difficulty;     // ← 注入
    ...
}

// 用的时候直接读字段(不用 if 判难度):
int interval = Math.max(difficulty.minSpawn, difficulty.spawnInterval - getElapsedSec());
boolean brute = getElapsedSec() >= difficulty.bruteMinSec && random.nextDouble() < difficulty.bruteChance;
player.takeDamage(difficulty.hitDamage);
String title = "打僵尸 [" + difficulty.label + "]";
```

### 6. 「难度随时间递增」是怎么实现的

注意 `spawnInterval`(初始)和 `minSpawn`(封顶)是两个值:

```java
int interval = Math.max(difficulty.minSpawn, difficulty.spawnInterval - getElapsedSec());
```

- 刚开始(`elapsedSec=0`):interval = spawnInterval(简单 90、困难 60)。
- 越打越久:`spawnInterval - elapsedSec` 越来越小 → interval 越来越小 → **刷怪越来越密**。
- 但有 `Math.max(minSpawn, ...)` 兜底:不会无限小,最低 `minSpawn`(简单 30、困难 20)封顶,**不会失控**。

> 这是「**档内递增**」:选了简单档,档位本身固定,但档位内随时间越来越难、有上限。和「简单/困难」是「**档位**」两回事——本项目有**两层难度设计**(选档 + 档内递增)。

### 7. `public final` 字段的设计

```java
public final int spawnInterval;     // public + final
```

- `public`:外面(`GameController`)直接 `difficulty.spawnInterval` 读,不用写 getter——简洁。
- `final`:构造后就不可改,**保证难度参数在游戏过程中不会被改动**(否则有人中途改 spawnInterval 就乱套了)。
- 这是 enum 字段的常见写法:**公开只读**(public final = "可以读、不能改")。

### 小自测

1. **enum 和普通常量(`static final int`)比,有什么好处?**
2. **`EASY(90,30,10,0.18,20,"简单")` 这行在干嘛?**
3. **为什么 Difficulty 的字段是 `public final`?**
4. **难度「随时间递增」靠哪两个字段?公式是什么?为什么不会失控?**
5. **加第三档「地狱」难度,要改哪些地方?(提示:不用改 GameController)**
6. **GameController 里为什么不用 `if(difficulty==EASY)...else...`?**

<details><summary>答案</summary>

1. enum **类型安全**——`Difficulty d = Difficulty.EASY` 一目了然,编译器检查合法值;普通 int 常量(5)看不出含义、易写错。而且 enum 能带字段/方法(本项目就用它带参数)。
2. 在调 Difficulty 的**构造器**,把 EASY 这个常量的 6 个字段填成 `(90,30,10,0.18,20,"简单")`。
3. `public` = 外面能直接读(不用 getter,简洁);`final` = 构造后不可改(难度参数运行中不能被改)。
4. `spawnInterval`(初始)和 `minSpawn`(封顶)。公式 `interval = max(minSpawn, spawnInterval - elapsedSec)`——越打 interval 越小(刷越密),但 `max(minSpawn,...)` 兜底不会低于 minSpawn,所以不失控。
5. 只要在 enum 加 `HELL(...)` 一个常量 + MainFrame 难度弹窗加选项。**GameController 不用改**(它读 `difficulty.xxx`,自动用新档的值)——这正是参数化的好处。
6. 因为参数已封装进 enum 字段,代码只读 `difficulty.spawnInterval` 等,**同一套代码跑不同数值**,不用 if 分档。if-else 写法是反例(每加参数就要改 if)。

</details>

## 二、`GameController` —— 游戏大脑 ⭐⭐

> 文件:[`game/GameController.java`](../../src/main/java/com/game/game/GameController.java)(~468 行,整个游戏最核心的类)
> 一句话职责:**不含 Swing Timer**(便于脱离界面单测),只管"一帧该干什么"——被 `GamePanel` 的 Timer 每 16ms 调一次 `update()` 推进一帧。

### 字段一览(分组)

> **关键设计**:GameController **不持有任何 Swing 组件**(没 Timer、没 JFrame)——纯逻辑。这样它能脱离界面跑单元测试(`TestGameLogic` 直接 `new GameController()` 调 update)。这就是 03 讲的「**关注点分离**:逻辑和界面分开」。

### 构造方法:初始化一局(注入难度)

> 📍 [`GameController(Difficulty)` 第 78–94 行](../../src/main/java/com/game/game/GameController.java#L78)

```java
public GameController() {                    // 无参:默认简单档(给 TestGameLogic 等无界面测试用)
    this(Difficulty.EASY);
}

public GameController(Difficulty difficulty) {
    if (difficulty == null) {                 // ① 防御:难度不能传 null
        throw new IllegalArgumentException("难度不能为空");
    }
    this.difficulty = difficulty;             // ② 注入难度(存字段,之后 update/shoot 都读它)
    this.player = new Player(WIDTH / 2.0, HEIGHT / 2.0);  // ③ 玩家放画布正中
    this.zombies = new ArrayList<>();         // ④ 四个空列表(僵尸/子弹/粒子/飘字)
    this.bullets = new ArrayList<>();
    this.particles = new ArrayList<>();
    this.floatingTexts = new ArrayList<>();
    this.random = new Random();               // ⑤ 随机数生成器(刷怪位置/概率用)
    this.score = 0;                           // ⑥ 初始状态:分数/击杀清零
    this.killCount = 0;
    this.running = true;                      //    游戏开始跑
    this.frameCounter = 0;                    //    帧计数归零(计时从 0 开始)
    this.gameOverFired = false;               //    死亡回调还没触发过
}
```

逐行:

- **① null 检查 + 抛异常**:**fail-fast**(快速失败)——难度传 null 是调用方的 bug,不检查的话后面 `difficulty.spawnInterval` 会抛 `NullPointerException`(更难定位)。立刻抛 `IllegalArgumentException` + 明确消息,让错误**在发生时就暴露**。
- **② `this.difficulty = difficulty`**:这就是上一节 Difficulty 讲的**注入**——把外面选好的难度存进字段,之后 `update`/`spawnZombie`/`handleCollisions` 都读 `difficulty.xxx`。
- **③ `new Player(WIDTH/2, HEIGHT/2)`**:玩家从画布**正中**出生(400, 300)。
- **④ 四个空 `ArrayList`**:开局什么都没有(没僵尸、没子弹、没特效)。后续 update 往里加、`removeIf` 往外清。
- **⑤ `new Random()`**:Java 的随机数工具,刷怪位置 / 概率判定用它。
- **⑥ 初始状态**:分数 0、击杀 0、`running=true`(游戏在跑)、`frameCounter=0`(计时从 0 开始)、`gameOverFired=false`(没死过)。

> **两个构造器的关系(重载)**:`GameController()`(无参)调 `this(Difficulty.EASY)`——默认简单档,给 `TestGameLogic` 等无界面测试用;`GameController(Difficulty)` 是真正干活的。这是**构造器重载**(同名不同参数,见 [Java方法签名与Javadoc](../Java方法签名与Javadoc.md))。

### `update()` —— 一帧的 6 步主循环 ⭐⭐⭐(最核心)

> 📍 [`update()` 第 139–193 行](../../src/main/java/com/game/game/GameController.java#L139)

整个游戏的"心跳",每帧调一次:

```java
public void update() {//每一帧调用一次
    if (!running) return;                    // 游戏已停(死亡)就不再推进

    // 0) 应用移动输入(WASD):斜走归一化 + 钳制在画布内
    if (dx != 0 || dy != 0) {
        double len = Math.hypot(dx, dy);      // ← 归一化:避免对角线超速
        double vx = dx / len * Player.SPEED;
        double ny = clamp(player.getY() + vy, radius, HEIGHT - radius);  // ← 钳制:不出界
        player.setX(nx); player.setY(ny);
    }
    // 1) 推进所有对象
    player.update();                          // 玩家(衰减红闪)
    for (Zombie z : zombies) { z.setTarget(px,py); z.update(); }  // 僵尸先设目标再移动
    for (Bullet b : bullets) { b.update(); }  // 子弹飞
    // 1.5) 推进特效 + 清理已消亡的特效
    particles.removeIf(Particle::isDead);     // ← 用 removeIf 清理
    updateShake();  if (damageFlashFrames>0) damageFlashFrames--;
    // 2) 按节奏刷怪(难度随时间递增)
    frameCounter++;
    int interval = Math.max(difficulty.minSpawn, difficulty.spawnInterval - getElapsedSec());
    if (frameCounter % interval == 0) spawnZombie();
    // 3) 碰撞检测(只改状态,不增删集合!)
    handleCollisions();
    // 4) 清理失效对象(出界/已死的子弹、已死的僵尸)
    bullets.removeIf(b -> b.isDead() || b.isOffscreen(WIDTH, HEIGHT));
    zombies.removeIf(Zombie::isDead);
    // 5) 结束判定
    if (player.getHp() <= 0) { running = false; fireGameOver(); }
}
```
```text
每一帧
  │
  ├─ 0) 处理输入 → 移动玩家
  ├─ 1) 推进状态 → 玩家 / 僵尸 / 子弹
  ├─ 1.5) 推进特效 → 粒子 / 浮动文字 / 震屏 / 红屏
  ├─ 2) 刷怪逻辑 → 随时间加速
  ├─ 3) 碰撞检测 → 只改状态
  ├─ 4) 清理集合 → 移除失效对象
  └─ 5) 结束判定 → 玩家死亡则停
```
**两个最关键的工程点 ⭐:**

**① 第 0 步「归一化」**:`dx/len * SPEED`。如果直接 `dx * SPEED`,对角线移动(WASD 同时按)会比直线快 √2 倍。除以 `len`(向量长度)归一化成单位方向,再乘 SPEED——**斜走和直走一样快**。

**② 第 3 步「只改状态」vs 第 4 步「removeIf 清理」分开** ⭐⭐:碰撞检测时**遍历** zombies/bullets,如果边遍历边删(for 里 `zombies.remove(...)`),会抛 `ConcurrentModificationException`。所以 `handleCollisions` 里只**标记**(`z.takeDamage`、`b.setDead(true)`),等遍历完了,第 4 步用 `removeIf` 一次性清理被标记的。**这是 Java 集合遍历删除的经典套路。**

### `shoot()` —— 开火

> 📍 [`shoot()` 第 123–133 行](../../src/main/java/com/game/game/GameController.java#L123)

```java
public void shoot() {
    double angle = player.getAngle();         // 瞄准角度(鼠标方向)
    double muzzle = player.getRadius() + 8;   // 枪口离玩家中心
    double bx = player.getX() + Math.cos(angle) * muzzle;   // 枪口位置
    double by = player.getY() + Math.sin(angle) * muzzle;
    double vx = Math.cos(angle) * BULLET_SPEED;              // 子弹速度向量
    double vy = Math.sin(angle) * BULLET_SPEED;
    bullets.add(new Bullet(bx, by, vx, vy));
    spawnMuzzleFlash(bx, by, angle);          // 枪口火花特效
    SoundUtil.shoot();                        // 开火音效
}
```
```text
        枪口 (bx, by)
          ●
         ↗
        /  ← 偏移距离 (muzzle = radius + 8)
       /
      ● 玩家中心 (player.getX(), player.getY())
         radius
```
- **枪口位置** = 玩家中心 + 朝向 × (半径+8)。`cos/sin` 把角度拆成 x/y 分量。
- **子弹速度** = `cos/sin × BULLET_SPEED`(8 像素/帧),方向 = 瞄准方向。
- 一颗子弹 = 一个 `Bullet` 对象,加进 `bullets` 列表。

### `spawnZombie()` —— 刷怪(难度递增 + Brute + 安全距离)⭐

> 📍 [`spawnZombie()` 第 199–236 行](../../src/main/java/com/game/game/GameController.java#L199)

三个设计点:

**(a) Brute 判定**(L200):
```java
boolean brute = getElapsedSec() >= difficulty.bruteMinSec && random.nextDouble() < difficulty.bruteChance;
```
存活满 `bruteMinSec` 秒(简单 10、困难 5)后,按 `bruteChance` 概率(简单 18%、困难 35%)刷壮汉。前期只有普通僵。

**(b) 安全距离重试**(L206-233)⭐:
```java
double safeDist = player.getRadius() + zr + 40;   // 离玩家至少这么远
for (int attempt = 0; attempt < 10; attempt++) {
    // 随机选一条边 + 随机位置生成
    ...
    if (Math.hypot(x-px, y-py) >= safeDist) break; // 够远就用这个点
}
```
**为什么**:贴墙的玩家如果僵尸生成在身边,第一帧就撞到扣血——不公平(看不到、躲不开)。所以生成点必须离玩家 ≥ safeDist,最多重试 10 次。

**(c) 速度按类型**(L201):

### `handleCollisions()` —— 圆形碰撞 ⭐

> 📍 [`handleCollisions()` 第 241–285 行](../../src/main/java/com/game/game/GameController.java#L241)

两种碰撞,都用**圆心距 < 半径和**判定:

**① 子弹 × 僵尸:**
```java
if (Math.hypot(bx-zx, by-zy) < b.getRadius() + z.getRadius()) {
    z.takeDamage(1);          // 僵尸扣血
    b.setDead(true);          // 子弹标记删除(第 4 步清理)
    if (z.isDead()) {         // 打死了
        killCount++;
        score += z.getScoreValue();   // +10 / +25
        floatingTexts.add(... "+10" ...);  // 飘分
    }
    break;                    // ⭐ 一颗子弹只命中一只,退出内层循环
}
```

**② 僵尸 × 玩家:**
```java
if (Math.hypot(zx-px, zy-py) < z.getRadius() + player.getRadius()) {
    player.takeDamage(difficulty.hitDamage);  // 扣血(20/25)
    z.kill();                 // ⭐ 僵尸消失但不计分(防反复扣血)
    triggerShake(14, 6);      // 震屏
    damageFlashFrames = 20;   // 红屏
}
```

> **`break` 的作用**:子弹命中一只僵尸后立刻退出内层循环,不再检测其他僵尸——**一枪一只**。删掉 break 就变穿透弹。
> **`z.kill()` vs `z.takeDamage`**:撞玩家用 `kill()`(hp=0 但不计分),被子弹打死用 `takeDamage`(计分)。详见 [详细代码1 Zombie](./05_代码精读_详细代码1.md)。

### `fireGameOver()` + `getElapsedSec()`

```java
private void fireGameOver() {               // 死亡回调,只触发一次
    if (gameOverFired) return;
    gameOverFired = true;
    SoundUtil.gameOver();
    if (onGameOver != null) onGameOver.run();  // → GameWindow.settle(结算存档)
}
public int getElapsedSec() {
    return frameCounter / FRAMES_PER_SEC;    // 60 帧 = 1 秒
}
```

> **`Runnable` 是什么(简单说)**:Java 的一个接口,代表「**一段可以回头跑的代码**」。`GameController` 不管玩家死后该干啥(它不碰界面),所以让 `GameWindow` 把"结算存档"那段代码**塞进 `onGameOver`**;玩家死时调 `onGameOver.run()` 就执行它。相当于「**你给我一段代码,时机到了我帮你跑**」——这就是**回调(callback)**。塞代码时一般用 lambda:`controller.setOnGameOver(() -> settle())`。

- **`gameOverFired`**:防止死亡回调被重复触发(比如同一帧多次调用 update)。
- **`getElapsedSec`**:基于帧计数,**暂停不计入**(暂停时 update 不跑、frameCounter 不增)。

### 小自测

1. **为什么 update 第 0 步要 `dx/len*SPEED`?不除 len 会怎样?**
2. **为什么碰撞(handleCollisions)只标记、不直接 remove?**
3. **一颗子弹能打几只僵尸?靠什么实现?**
4. **僵尸撞玩家后为什么用 `kill()` 而不是 `takeDamage()`?**
5. **GameController 为什么不持有 Swing Timer?**
6. **暂停时时间为什么自动停?(提示:getElapsedSec)**

<details><summary>答案</summary>

1. 归一化。不除 len,对角线(WASD 同按)速度是 √2 倍直线,斜走超速。除 len 成单位方向再乘 SPEED,斜走直走一样快。
2. for 遍历集合时不能增删元素,否则 `ConcurrentModificationException`。所以只标记(takeDamage/setDead),遍历完用 removeIf 清理。
3. 一只。`break` 退出内层循环,不再检测其他僵尸。
4. `kill()` = hp=0 但不计分;`takeDamage` 打死才计分。僵尸撞玩家后必须消失(防反复扣血),但不是被你打死的,不给分。
5. 为了**关注点分离 + 可测试性**:不持有 Swing 组件,能脱离界面单测(`new GameController()` 调 update)。Timer 在 GamePanel 里。
6. `getElapsedSec = frameCounter/60`;暂停时 update 不被调、frameCounter 不增 → 秒数自动停。读系统时间做不到这点。

</details>
