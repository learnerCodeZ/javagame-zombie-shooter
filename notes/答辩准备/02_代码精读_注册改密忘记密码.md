## 开篇:这三件套和登录是同一个模子

01 里登录的流程是:

```
UI 取值 → 非空/格式校验 → 调 DAO → 据返回(true/false/null)弹框/跳窗
```

注册 / 改密 / 忘记密码**完全一样**,只是:校验项不同、DAO 方法不同、成功/失败的提示不同。所以下面三节你重点看**差异**,骨架不用重学。读完你会更确信:这套模式能套到任何"表单→写库"的功能上。

# §1 注册

## 1.1 `RegisterFrame.doRegister`(UI 层)

> 文件:[`ui/RegisterFrame.java` · `doRegister` 第 104–157 行](../../src/main/java/com/game/ui/RegisterFrame.java#L104)
> 📍 关键行:L105-07 取值 · L109-18 手机号校验(空/`\d{11}`) · L119-23 密码≥6 · L124-28 昵称≤50 · **L130-32 昵称留空默认「玩家+后4位」** · **L140 调 `register`** · L148-52 成功回登录 · L153-56 失败提示

逐段:

- **取值**:`phone` / `password` / `nickname`。
- **校验**:手机号非空 + `\d{11}`;密码≥6;昵称≤50;**昵称留空 → 默认「玩家+手机号后4位」**(避免一串数字当昵称难看)。
- 构造 `User` → 调 `register`。
- 成功:弹「注册成功」→ `dispose()` → 开 `LoginFrame`。失败:弹「注册失败(手机号可能已存在)」。

## 1.2 `UserDao.register`(DAO 层)

> 文件:[`dao/UserDao.java` · `register` 第 48–64 行](../../src/main/java/com/game/dao/UserDao.java#L48)
> 📍 关键行:**L49 先 `findByPhone` 查重** · L53 `INSERT` SQL · L56 手机号 · **L57 密码 MD5** · L58 昵称 · L59 `executeUpdate`

```java
public boolean register(User user) {
    if (findByPhone(user.getPhone())) { ...; return false; }   // ① 先查重
    String sql = "INSERT INTO user(phone, password, nickname) VALUES (?, ?, ?)";
    ... ps.setString(2, MD5Util.md5(user.getPassword()));      // ② 密码先 MD5 再存
    return ps.executeUpdate() > 0;                             // ③ 影响行数>0 = 成功
}
```

- **查重**:先 `findByPhone` → 已存在直接返回 false(**不写库**)。避免直接 INSERT 撞 `UNIQUE` 约束、被 catch 吞成模糊的「注册失败」。
- **INSERT**:3 个参数(手机号 / MD5 密码 / 昵称)。`role` 不写 → 取库默认 `'user'`(普通用户注册直接成功,不需审核)。
- **返回**:`executeUpdate() > 0` —— INSERT 成功影响 1 行 → true。

## 1.3 想改 X 动哪里 / 自测

- 想加「注册要填邮箱」:`doRegister` 加校验 + `register` 的 SQL 加列 + `User` 加字段 + DB 加列(横切,一路改)。
- **Q**:为什么先 `findByPhone` 再 INSERT,不直接 INSERT 让 UNIQUE 报错?

<details><summary>答案</summary>

直接 INSERT 撞 UNIQUE 会抛 `SQLException`,被 catch 吞成 `return false` → 界面弹「注册失败(手机号可能已存在)」——提示模糊(也可能是 DB 真的连不上)。先 `findByPhone` 查重能给出**准确的「手机号已存在」**,且**不依赖异常做流程控制**(异常应处理意外,不该当正常分支)。

</details>

# §2 修改密码

## 2.1 `ChangePasswordDialog.doSubmit`(UI 层)

> 文件:[`ui/ChangePasswordDialog.java` · `doSubmit` 第 105–145 行](../../src/main/java/com/game/ui/ChangePasswordDialog.java#L105)
> 📍 关键行:L106-08 取三个密码 · L110-14 非空 · **L116 两次新密码一致** · L122 新密码≥6 · **L130 调 `changePassword(id, old, new)`** · L137-40 成功 · L141-44 旧密码错误

- 注意:这个是 **`JDialog`(模态对话框)**,不是 `JFrame`。依附于主菜单窗,弹出后主菜单不可点(模态)。
- 校验:三项非空 → 两次新密码一致 → 新密码≥6。
- 调 `changePassword(当前用户 id, 旧密码, 新密码)`。
- 成功:弹「修改成功」+ `dispose`。失败:弹「原密码错误」(DAO 返回 false = 旧密码不对)。

## 2.2 `UserDao.changePassword`(DAO 层)⭐ 巧妙点

> 文件:[`dao/UserDao.java` · `changePassword` 第 110–122 行](../../src/main/java/com/game/dao/UserDao.java#L110)
> 📍 关键行:**L111 SQL:`UPDATE user SET password=? WHERE id=? AND password=?`** · L114 新密码 MD5 · L115 userId · **L116 旧密码 MD5 放进 WHERE**

```java
String sql = "UPDATE user SET password = ? WHERE id = ? AND password = ?";
ps.setString(1, MD5Util.md5(newPwd));   // 新密码(要改成的)
ps.setInt(2, userId);
ps.setString(3, MD5Util.md5(oldPwd));   // 旧密码(塞进 WHERE 校验)
return ps.executeUpdate() > 0;
```

- **巧妙**:旧密码校验**不在 Java 里 `if` 判断**,而是**塞进 WHERE**——`WHERE id=? AND password=MD5(旧密码)`。旧密码不对 → WHERE 匹配 0 行 → UPDATE 影响 0 行 → 返回 false。
- **好处**:一条 SQL **同时完成「校验旧密码 + 改新密码」**,不用先 SELECT 再 UPDATE(两步之间有并发间隙),也少一次 DB 往返。
- **返回**:`executeUpdate() > 0` —— 1 行 = 旧密码对且已改;0 行 = 旧密码错。

## 2.3 想改 X 动哪里 / 自测

- 想让「新密码不能和旧密码相同」:`doSubmit` 里加 `if (newPwd.equals(oldPwd))` 提示。
- **Q**:为什么旧密码校验放 WHERE,不先 SELECT 比对再 UPDATE?

<details><summary>答案</summary>

一条 SQL **原子**完成,避免「SELECT 时旧密码对、UPDATE 前被人改了」的并发间隙;也少一次 DB 往返。这是「用 SQL 的 WHERE 顺带做校验」的典型手法,比"先查后改"更稳更快。

</details>

# §3 忘记密码(跨 DAO 的申请流程)⭐ 最复杂

这个和前两个不一样:它**跨两个 DAO**(`UserDao` 查手机号 + `ResetRequestDao` 提交申请),而且是**异步流程**(用户提交 → 等 admin 审核 → 重置)。

## 3.1 流程全景

```
玩家(登录页)                    admin(用户管理页)
  │ 提交申请                        │
  ▼                                 │
LoginFrame.doResetRequest           │
  → UserDao.findByPhone  (查手机号存不存在)
  → ResetRequestDao.requestReset (插一条 pending 申请)
        status=pending ──────────────┐
                                       │  玩家等审核
  玩家用 123456 登录 <────── admin 看到 pending,点「通过」
                                       │
                            ResetRequestDao.approve
                              → 事务:UPDATE user 密码=MD5('123456')
                                       + UPDATE 申请 status=approved
```

## 3.2 `LoginFrame.doResetRequest`(UI 层)

> 文件:[`ui/LoginFrame.java` · `doResetRequest` 第 154–195 行](../../src/main/java/com/game/ui/LoginFrame.java#L154)
> 📍 关键行:L155 弹框输入手机号 · L157-60 取消 · L162-66 非空 · L167-71 `\d{11}` · **L176 `findByPhone` 区分「手机号不存在」** · **L181 `requestReset` 提交** · L182-88 据结果弹框

- 用 `JOptionPane.showInputDialog` 弹个输入框(不是表单窗体)。
- 校验:非空 + `\d{11}`。
- **两步调 DAO**:先 `findByPhone`(区分「手机号不存在」),再 `requestReset`(提交申请)。
- `requestReset` 返回 false = 该用户已有 pending 申请(查重)→ 提示「已有待审核申请」。

## 3.3 `ResetRequestDao.requestReset`(提交申请)

> 文件:[`dao/ResetRequestDao.java` · `requestReset` 第 32–63 行](../../src/main/java/com/game/dao/ResetRequestDao.java#L32)
> 📍 关键行:L37 `SELECT id WHERE phone`(查 userId) · L45 手机号不存在→false · **L49 `hasPending` 查重** · L53-55 INSERT pending 申请

```java
public boolean requestReset(String phone) {
    // 1) 查 userId
    SELECT id FROM user WHERE phone = ?  → userId
    if (userId == null) return false;        // 手机号不存在
    // 2) 已有 pending 申请则不重复
    if (hasPending(userId)) return false;    // 查重
    // 3) 插一条 pending 申请
    INSERT INTO password_reset_request(user_id, status, request_time) VALUES (?, 'pending', NOW())
}
```

- **`hasPending`**(L71-86):`SELECT COUNT(*) WHERE user_id=? AND status='pending'` → 防同一用户重复堆申请。
- 申请状态 `pending` = 待审。

## 3.4 `ResetRequestDao.approve`(admin 审核 → 重置)⭐ 事务

> 文件:[`dao/ResetRequestDao.java` · `approve` 第 120–161 行](../../src/main/java/com/game/dao/ResetRequestDao.java#L120)
> 📍 关键行:**L124 `setAutoCommit(false)` 开事务** · L128 `UPDATE user` 密码=MD5('123456') `WHERE id=? AND role='user'` · L131 `reset=影响行数>0` · L134-40 按结果置申请 approved/rejected · **L141 `commit`** · catch L147 `rollback` · finally 还原+close

```java
public boolean approve(int requestId, int userId) {
    conn.setAutoCommit(false);                          // ① 开事务
    // ② 重置密码(仅 role='user')
    UPDATE user SET password=MD5('123456') WHERE id=? AND role='user'
    reset = executeUpdate() > 0;                        // admin 不可重置 → 0 行
    // ③ 按结果置申请状态
    newStatus = reset ? "approved" : "rejected";
    UPDATE password_reset_request SET status=?, handle_time=NOW() WHERE id=?
    conn.commit();                                      // ④ 提交
    return reset;
    // catch: rollback; finally: setAutoCommit(true) + close
}
```

- **事务**:两步 UPDATE(改密码 + 改申请状态)**要么都成、要么都废**——不能「密码改了但申请状态没更新」(否则同一条申请能被重复审核)。
- **admin 保护**:`WHERE role='user'` —— admin 账号影响 0 行 → `reset=false` → 申请置 `rejected`(超级账号不可被重置)。
- **commit 条件**:这里**无条件 commit**(两步都执行了就提交);`reset` 真假只决定申请状态(approved/rejected),不影响提交。(对比 `deleteUser` 的 `if(deleted) commit() else rollback()`——那是另一处事务,commit 条件不同,见 [审计修复②](../v1.0_审计修复/扩展_审计修复_数据与玩法.md)。)

## 3.5 想改 X 动哪里 / 自测

- 想让重置密码改成随机串:`approve` 里把 `MD5('123456')` 换成随机生成 + 邮件发送(课设范围外)。
- **Q1**:为什么 `approve` 要包事务,而 `register` / `changePassword` 不用?
- **Q2**:`hasPending` 查重和 INSERT 不在同一事务,有竞态吗?

<details><summary>答案</summary>

**Q1**:`approve` 是**两步写**(改密码 + 改申请状态),要保证一致性,所以包事务;`register` / `changePassword` 只**一步写**,autocommit 即可。**事务用在「多步写必须同时成功」的场景**。

**Q2**:有,理论 TOCTOU——`hasPending`(独立连接)查完再 INSERT,并发下可能重复插。课设单用户本地触发不了;生产应并同一连接 + `SELECT...FOR UPDATE` 或给 `(user_id)` 加唯一约束。这是 [已知问题 #6](../../docs/已知问题与后续优化.md)。

</details>

# §4 三件套 vs 登录:一张表收尾

| 功能 | 校验 | DAO 方法 | SQL | 返回 | 成功 / 失败 |
|---|---|---|---|---|---|
| 登录 | 非空 + `\d{11}` | `login(phone, pwd)` | `SELECT WHERE phone AND password` | `User`/`null` | 进菜单 / 报错 |
| 注册 | 非空 + `\d{11}` + 密码≥6 + 昵称≤50 | `register(user)` | `INSERT`(先 `findByPhone` 查重) | `boolean` | 回登录 / 手机号已存在 |
| 改密 | 三项非空 + 两次一致 + 新密≥6 | `changePassword(id, old, new)` | `UPDATE WHERE id AND password=旧` | `boolean` | 改成功 / 原密码错误 |
| 忘记密码 | 非空 + `\d{11}` | `findByPhone` + `requestReset` | `SELECT id` + `INSERT pending` | `boolean` | 已提交 / 不存在 / 已有 pending |

**共同骨架**:UI 取值 → 校验 → 调 DAO → 据返回弹框。差异只在**校验项、SQL、返回语义**。看懂登录,这三个就是举一反三。
