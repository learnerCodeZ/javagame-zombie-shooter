## 一、`Player` —— 玩家(`draw` + `takeDamage`)

> `Player.draw()` 是 `GameObject` 接口 `draw()` 的实现(`@Override`),每帧由 `GamePanel.paintComponent` 调一次,负责"把玩家画到画布上"。一个方法里藏了 **4 个干货技巧**,逐个拆。

### 方法全貌(先看结构,再看细节)

```java
@Override
public void draw(Graphics2D g) {
    int ix = (int) x;                       // ① 坐标取整
    int iy = (int) y;
    BufferedImage sp = sprite();            // ② 拿精灵图(懒加载,首次读盘后缓存)

    if (sp != null) {                       // ③ 图读到了 → 贴图路线
        // ... 贴图 + 朝向镜像 + 受伤红闪
    } else {                                // ④ 图读不到 → 回退画橙猫兜底
        // ... 用代码画
    }
}
```

**两条路线**:图读到了就贴图(主路径);读不到(null)就用代码画个橙猫,保证不崩。这是**回退兜底**思想——外部资源(图片)不可用时,程序仍能跑、仍能看。

---

### 主路径(贴精灵图)逐行精讲 ⭐

### ② `BufferedImage sp = sprite();`

调 `sprite()` 拿精灵图(详见 [03 §2.1](./03_代码精读_游戏核心.md) 的懒加载讲解):第一次调时从磁盘读 `player.png` 进内存,之后每帧从内存取(极快)。读失败返回 `null`。

### 贴图 + 水平镜像(核心)⭐

```java
int sw = sp.getWidth();                     // 原图宽(像素)
int sh = sp.getHeight();                    // 原图高
boolean faceLeft = Math.cos(angle) < 0;     // 朝左? angle 是弧度,cos<0 = 鼠标在玩家左半边
if (faceLeft) {
    g.drawImage(sp, ix + radius, iy - radius, ix - radius, iy + radius, 0, 0, sw, sh, null);
} else {
    g.drawImage(sp, ix - radius, iy - radius, ix + radius, iy + radius, 0, 0, sw, sh, null);
}
```

#### 先搞懂 10 参数 `drawImage` 在干嘛

```java
g.drawImage(img, dx1, dy1, dx2, dy2,  sx1, sy1, sx2, sy2,  observer)
```

一句话:**把"源图里的一块矩形"缩放着画到"画布上的一块矩形"里**。

- 后四个 `sx1,sy1,sx2,sy2` = **源图**里的矩形(两个角)。本例 `0,0,sw,sh` = 整张图。
- 中间四个 `dx1,dy1,dx2,dy2` = **画布**上的矩形(两个角)。
- ⭐ 关键:`(dx1,dy1)` 和 `(dx2,dy2)` 是**两个角**(不是"左上角+宽高"),而且**按顺序对应源图**——源图**左上角 `(sx1,sy1)`** 画到 **`(dx1,dy1)`**,源图**右下角 `(sx2,sy2)`** 画到 **`(dx2,dy2)`**。
- 最后一个 **`observer`**(`ImageObserver`)= 图片**异步加载**完成时的回调;这里传 `null`,因为 `sp` 是 `BufferedImage`(`ImageIO.read` 同步全量加载、画的时候像素早全好了),没东西可通知。只有用老的异步加载(`Toolkit.getImage`)才需要传一个 observer(如组件 `this`)让它"图到了叫我重画"。

#### 代入数字看(假设 ix=400, iy=300, radius=26, 图 128×128)

**else 分支(朝右,正常):**

```java
g.drawImage(sp, ix-radius, iy-radius,   ix+radius, iy+radius,   0, 0, sw, sh, null);
//              374      274             426      326
```

画布框 = (374,274)~(426,326),一个 52×52、中心在玩家 (400,300) 的方框。对应关系:

```
源图左上角 (0,0)     ──画到──▶  画布 (374,274)  ← 左上
源图右下角 (128,128) ──画到──▶  画布 (426,326)  ← 右下
```

源图**左边**→画布**左边**、**右边**→画布**右边** → 图**正常方向**显示(玩家脸朝右)。

**if 分支(朝左,x 对调了!):**

```java
g.drawImage(sp, ix+radius, iy-radius,  ix-radius, iy+radius,  0, 0, sw, sh, null);
//              426      274            374      326     ← 第一个 x=426(右)、第二个 x=374(左)
```

x 坐标**对调**了!对应关系变成:

```
源图左上角 (0,0)     ──画到──▶  画布 (426,274)  ← 右上!!
源图右下角 (128,128) ──画到──▶  画布 (374,326)  ← 左下!!
```

源图**左边**→画布**右边**、**右边**→画布**左边** → 图**左右翻过来**(镜像)→ 玩家脸朝左!

#### 画个图对照

```
源图 player.png(默认画的是朝右 →):
┌───────────────┐
│        →  朝右  │   左边 x=0          右边 x=128
└───────────────┘

朝右(else):                  朝左(if,x 对调):
源图左→画布左、右→画布右       源图左→画布右、右→画布左
┌───────────────┐            ┌───────────────┐
│        →  朝右  │            │  ←  朝左       │   ← 镜像了!
└───────────────┘            └───────────────┘
   (正常)                       (水平翻转)
```

#### 一句话记

**目标矩形的 x 两个坐标一对调(`ix+radius` 在前、`ix-radius` 在后),图就水平镜像。** 不用翻转图片像素,换坐标顺序就行。y 没对调(还是 `iy-radius` 在前、`iy+radius` 在后)→ 只左右翻、不上下翻。Java 2D 常用小技巧:**目标 x 对调 = 水平翻转**。

> **朝向判断 `Math.cos(angle) < 0`**:`angle` 是枪口角度(弧度,由鼠标方向 `atan2(dy,dx)` 算出)。`cos(angle)` 是方向的**水平分量**:cos>0 朝右、cos<0 朝左、cos=0 正上/正下(归到 else 当朝右处理)。所以 `cos<0` ≈ 朝左 → 走镜像分支。

### 受伤红闪(SRC_ATOP 局部染色)⭐⭐

```java
if (hitFlash > 0) {                                  // 受伤了(hitFlash>0,每帧衰减)
    float a = Math.min(1f, hitFlash / 10f) * 0.5f;   // 透明度:随 hitFlash 衰减,最大 0.5
    Composite old = g.getComposite();                 // ① 先保存当前合成规则
    g.setComposite(AlphaComposite.getInstance(AlphaComposite.SRC_ATOP, a));  // ② 换成 SRC_ATOP
    g.setColor(new Color(255, 60, 60));               // 红色
    g.fillRect(ix - radius, iy - radius, radius * 2, radius * 2);  // ③ 在角色区域填红
    g.setComposite(old);                              // ④ 还原合成规则(别影响后续绘制)
}
```

#### 先理解「合成规则(Composite)」是什么

画布上每个像素,你新画一笔时,新颜色要和**原来已有的颜色**混合。**怎么混合**由「合成规则」(`Composite`)决定;**混合多少**由 alpha(透明度,0~1)决定。

- `g.setComposite(规则)` = 设置接下来怎么混合。
- 默认规则是 `SRC_OVER`:新颜色**直接盖**在旧颜色上(alpha 控制盖多透)。
- `AlphaComposite.getInstance(规则, alpha)` = 拿一个「指定规则 + 指定透明度」的合成对象。

#### 问题:直接 fillRect 红色会怎样?

精灵图 `player.png` 是**透明背景 PNG**——角色本体(橙猫)的像素不透明,四个角是**透明**的:

```
画布上贴好精灵图后(□=透明角,■=猫的不透明像素):
    □□□□■■■■■■□□□□
    □□■■■■■■■■■■□□        ← 中间是猫,四角透明
    □■■■■■■■■■■■■□
    □□■■■■■■■■■■□□
    □□□□■■■■■■□□□□
```

要"染红"。如果不管合成规则、直接 `fillRect` 一个红方块盖在 2×radius 方框上(默认 SRC_OVER),**红色会把整个方框盖住**,包括本来透明的四个角 → 画面出现一个**红方块**,角色被盖住、很难看。

我们要的只是「**猫本身变红**,透明角还是透明」。

#### 解法:`SRC_ATOP` —— 只染已有内容

`SRC_ATOP` 规则:**新颜色只应用在「目标已有不透明像素」的位置**,透明位置不动。

```
用 SRC_ATOP 染红:
    □□□■■(红)■■■■□□□      →  只有猫(■)变红,透明角(□)还是透明
    □□■■■■■■■■■■□□           画面 = 一只红猫,没有红方块 ✅
    ...
```

原理:角色本体像素不透明 → SRC_ATOP 让红盖上去;四个角透明(没有"已有内容")→ SRC_ATOP 不动它们。所以**填一个红方块,实际只染到了猫的形状**——等于拿猫自己的形状当了"蒙版"。

#### alpha `a` 怎么算出来的(红闪渐淡)

```java
float a = Math.min(1f, hitFlash / 10f) * 0.5f;
```

- `hitFlash` 从 10 衰减到 0(每帧 `--`,挨打时 `takeDamage` 设成 10)。
- `hitFlash / 10f` = **1.0 → 0.0**(刚挨打是 1,10 帧后归 0)。
- `× 0.5f` = **0.5 → 0.0**(上限压到 0.5)。
- `Math.min(1f, ...)` = 钳到 1.0 以内(防御:万一 hitFlash>10 也不超)。

所以 alpha 从 **0.5(刚挨打,半透明红)** 渐变到 **0(红消失)**——红闪**由浓转淡**。为什么上限 0.5 不是 1.0:1.0 会把猫整个换成纯红(看不见猫了);0.5 是**半透明红叠在猫上** = "猫偏红了"还能看出原样,更像"受伤泛红"。

#### 4 步套路:保存 → 切换 → 画 → 还原 ⭐

```java
Composite old = g.getComposite();                 // ① 保存当前规则(默认 SRC_OVER)
g.setComposite(AlphaComposite.getInstance(...));  // ② 切到 SRC_ATOP
g.setColor(...); g.fillRect(...);                 // ③ 在这个规则下画
g.setComposite(old);                              // ④ 还原成旧规则
```

**为什么要保存/还原**:`setComposite` 改的是**画布的全局状态**——改了之后**后面所有绘制**都会用新规则,直到你再改回去。这里只想让"染红这一下"用 SRC_ATOP,染完得立刻还原,否则之后画的子弹/僵尸/粒子全被 SRC_ATOP 影响、画面错乱。所以:**改全局状态前先存、用完立刻还原**——这是绘图里改任何全局状态(Composite/Color/Font/Stroke)的通用套路。

#### 对比:回退分支(画橙猫)为什么不用 SRC_ATOP?

`else` 分支(精灵图读不到、用代码画橙猫时)的红闪是这么写的:

```java
g.setColor(new Color(255, 60, 60, (int) (a * 255)));   // 直接用「带透明度的红色」
g.fillOval(ix - radius, iy - radius, radius * 2, radius * 2);  // 画一个红椭圆
```

它**没用 SRC_ATOP**,而是把透明度直接塞进 `Color`(第 4 个参数 `(int)(a*255)` = alpha 0~255)。为什么行?因为橙猫是用 `fillOval` 画的**实心椭圆**(椭圆本身就是猫的形状,没有"透明角"问题)——直接拿半透明红盖上去,自然只盖在椭圆上。而精灵图分支是贴的 PNG(有透明角),才需要 SRC_ATOP 来"只染不透明像素"。**两种做法目的一样:只染角色形状、不出现红方块**,只是绘制方式不同(贴图 vs 画椭圆)选了不同手段。

> 一句话:`SRC_ATOP` = 「新颜色只染已有内容」的合成规则,用来给**透明背景 PNG 的角色**局部染红(不出现红方块);配合 alpha 渐变实现"挨打由浓转淡"的红闪;用完要还原合成规则,别污染后续绘制。

### 回退路径(画橙猫,图读不到时)

`else` 分支(精灵图为 null):用 `fillPolygon`(耳朵)+ `fillOval`(身体/眼睛)+ `drawLine`(枪管)拼出一只橙色小猫。原理简单(基础绘图 API),关键是**思想**:

- 外部资源(图片)可能缺失/损坏 → 不能让游戏崩。
- `sprite()` 读失败返回 null → `draw` 走 else 用代码画兜底 → 玩家照样能看到角色、游戏照跑。
- 回退画法里**也带了红闪**(`if (hitFlash>0)` 那段,用带 alpha 的红色 fillOval),保持两条路线表现一致。

> 这是**防御式编程**:核心功能(贴图)失败时,有降级方案(代码画),而不是崩或空白。Zombie 的 draw 同款思路。

### `takeDamage(int dmg)` —— 受伤(触发红闪的那一半)

> 文件:[`Player.java` L130–136](../../src/main/java/com/game/game/Player.java#L130)
> 谁调它:`GameController.handleCollisions` 里僵尸撞到玩家时 `player.takeDamage(difficulty.hitDamage)`(简单 20、困难 25,来自 `Difficulty` 枚举)。

```java
public void takeDamage(int dmg) {
    this.hp -= dmg;            // ① 扣血
    if (this.hp < 0) {         // ② 钳到 0(不让血量变负)
        this.hp = 0;
    }
    this.hitFlash = 10;        // ③ 触发红闪(上面 draw 里读的那个)
}
```

逐行:

- **① `this.hp -= dmg`**:扣掉伤害值。`this.hp` = 当前这个玩家对象的血量字段(`this` 指当前对象,显式写出更清晰);`dmg` 是调用方传进来的伤害数。
- **② `if (this.hp < 0) this.hp = 0`**:**钳制(clamp)**到 0,不让血量变负。血量是血条,负数没意义(血条画不出负的);而且 `GameController` 用 `player.getHp() <= 0` 判死亡,负数也能触发,但钳到 0 更干净。
- **③ `this.hitFlash = 10`**:**触发受伤红闪**——把红闪计数器设成 10(帧)。这正好接上 §二 的「受伤红闪」:`draw()` 每帧读 `hitFlash`,>0 就用 `SRC_ATOP` 染红;`Player.update()` 每帧 `hitFlash--` 衰减,10 帧后归零、红闪消失。

> **「挨打 → 变红 → 渐渐恢复」的完整链条** = 三部分配合:
> `takeDamage` 设 `hitFlash=10`(起) → `update` 每帧 `hitFlash--`(衰减) → `draw` 读 `hitFlash` 染红(渲染)。

### takeDamage 和 draw 的关系(状态 vs 渲染)

`takeDamage` **只改状态**(扣血 + 设 hitFlash),**不画图**;`draw` 读 hitFlash 决定要不要染红。**状态和渲染分开**——这是游戏的常见分工:逻辑层改数据,渲染层读数据画。挨打时玩家不用自己"画红",只把 hitFlash 设上,draw 自然会读它染红。

> 一句话:「`takeDamage` = 扣血(钳到 0)+ 设 `hitFlash=10` 触发红闪;和 `draw` 配合完成『挨打变红』效果,状态归状态、画归画。」

## 二、`Particle` —— 粒子特效

> 文件:[`game/Particle.java`](../../src/main/java/com/game/game/Particle.java)(71 行,纯视觉特效)
> 谁创建/清理它:`GameController` 的 `spawnMuzzleFlash`(枪口火花)/`spawnHitSparks`(命中火花)/`spawnZombieDeath`(僵尸爆血)批量 `new` 出来丢进 `particles` 列表;每帧 `update` + `draw`,寿命尽了 `particles.removeIf(Particle::isDead)` 清掉。**用完即弃**。

### 字段一览(逐个注释)

```java
private double x, y;          // 位置(画布像素)
private double vx, vy;        // 速度(像素/帧)
private final Color color;    // 颜色(火花偏橙黄、血液偏绿、...)
private final int size;       // 基准半径(像素)
private int life;             // 剩余寿命(帧),每帧 -- 衰减
private final int maxLife;    // 初始寿命(构造时记一份,draw 算"剩余比例"用)
```

> `maxLife` 是 `final`(不变),`life` 每帧减 1。`draw` 里用 `life/maxLife` 算"还剩百分之几的命",据此淡出 + 缩小。

### `update()` —— 位移 + 阻尼 + 寿命递减

```java
public void update() {
    x += vx;        // ① 按速度移动
    y += vy;
    vx *= 0.92;     // ② 阻尼(每帧速度 ×0.92,模拟空气阻力 → 粒子越飞越慢)
    vy *= 0.92;
    life--;         // ③ 寿命 -1
}
```

- **① 位移**:和 Bullet 一样,按速度向量平移。
- **② 阻尼 `*= 0.92`**:每帧速度衰减 8% → 粒子飞着飞着就慢下来、最终停住。模拟"空气阻力"最简单的写法(乘个小于 1 的系数)。**Bullet 没有阻尼(匀速直线)**,Particle 有(减速)——子弹该一直飞,火花/血液飞溅后该逐渐减速。
- **③ `life--`**:寿命每帧 -1,归零就 `isDead`。

### `draw()` —— 按剩余寿命淡出 + 缩小

```java
public void draw(Graphics2D g) {
    float ratio = Math.max(0f, life / (float) maxLife);   // ① 剩余寿命比例 1.0→0
    int alpha = (int) (ratio * 255);                       // ② 透明度 = 比例 ×255
    if (alpha <= 0) return;                                // ③ 完全透明就不画
    g.setColor(new Color(color.getRed(), color.getGreen(), color.getBlue(), alpha)); // ④ 带 alpha 的颜色
    int s = Math.max(1, (int) (size * ratio));             // ⑤ 大小也按比例缩(最小 1)
    int ix = (int) x; int iy = (int) y;
    g.fillOval(ix - s, iy - s, s * 2, s * 2);             // ⑥ 画圆
}
```

- **① `ratio = life/maxLife`**:剩余寿命比例。刚生成 life=maxLife → ratio=1.0;快死 life→0 → ratio→0。
- **② `alpha = ratio*255`**:透明度跟寿命挂钩 → 粒子**越接近寿命终了越透明**(渐淡)。
- **③ `if(alpha<=0) return`**:完全透明就不画(省一次绘制)。
- **④ `new Color(r,g,b,alpha)`**:Color 的 **4 参数版**,第 4 个是 alpha(0~255)。注意这里**没用 SRC_ATOP**(对比第一部分 Player 的红闪)——粒子是直接画在画布上的一个圆,没有"已有内容"要避让,直接用带透明度的颜色就行。
- **⑤ `s = max(1, size*ratio)`**:大小也按比例缩 → 粒子**越接近寿命终了越小**(渐缩)。`max(1,...)` 保证最小 1 像素,不归 0。
- **⑥ `fillOval`**:画直径 2s 的圆。

> **渐淡 + 渐缩同时**:粒子从"大而清晰"逐渐变成"小而透明"最后消失——靠 `ratio` **一个变量同时驱动** alpha 和 size,自然消散。

### `isDead()` + 清理

```java
public boolean isDead() { return life <= 0; }
```

`life` 归零 → `isDead()` 真 → `GameController.update()` 里 `particles.removeIf(Particle::isDead)` 清掉。粒子**用完即弃**,不长期占内存。

### 设计点:Particle 为什么不实现 GameObject?⭐

`Particle` 有 `update()`/`draw()`,但**没有 `implements GameObject`**(对比 Player/Zombie/Bullet 都实现了)。为什么?

- `GameObject` 是给**参与游戏逻辑的实体**用的(要碰撞、要统一管理);Particle 是**纯视觉特效**(火花/血液),不参与碰撞、不影响玩法。
- 所以它独立存在,`GameController` 用**单独的 `particles` 列表**管理(不和 zombies/bullets 混),逻辑更清晰:**实体归实体、特效归特效**。
- 它的 `update/draw` 是"凑巧同名",不是接口强制——属于**鸭子类型**式的并行结构。

> 这是"接口不是万能"的例子:不是所有有 update/draw 的东西都该塞进同一个接口;按**职责**分组(实体 vs 特效)比强行统一更合理。

### 想改 X 动哪里

| 粒子更大 | 构造时传更大的 `size` |

## 三、`Zombie` —— 僵尸

> 文件:[`game/Zombie.java`](../../src/main/java/com/game/game/Zombie.java)
> 谁创建/清理:`GameController.spawnZombie()` 按「难度递增 + Brute 概率」生成,丢进 `zombies` 列表;每帧 `update` + `draw`,被打死或撞玩家后 `zombies.removeIf(Zombie::isDead)` 清掉。
> 和 Player 的关系:**都贴精灵图、都有 hitFlash 闪、都实现 GameObject**——但 Zombie 多了「两种类型」「追踪玩家」「头顶血条」「两种死法」。

### 字段一览(逐个注释)

```java
public enum Type { NORMAL, BRUTE }        // 两种僵尸:普通绿僵 / 壮汉

private static BufferedImage SPRITE;       // 精灵图(类级懒加载,和 Player 同款)
private final Type type;                    // 类型(构造时定,不变)
private double x, y;                        // 位置
private final double speed;                 // 速度(壮汉更慢)
private int hp;                             // 当前血量
private final int maxHp;                    // 满血(画血条比例用)
private final int radius;                   // 半径(碰撞 + 贴图大小)
private final int scoreValue;               // 击杀得分
private double targetX, targetY;            // 追踪目标(玩家位置,每帧由控制器回填)
private int hitFlash;                       // 命中闪白剩余帧
private int animFrame;                      // 动画帧(身体起伏摇摆)
private boolean facingLeft;                 // 朝向(贴图镜像用)
```

### 构造:按 `Type` 定属性 ⭐

```java
public Zombie(Type type, double x, double y, double speed, double targetX, double targetY) {
    this.type = type; ...
    if (type == Type.BRUTE) {
        this.radius = 26;  this.maxHp = 3;  this.hp = 3;  this.scoreValue = 25;
    } else {
        this.radius = 18;  this.maxHp = 1;  this.hp = 1;  this.scoreValue = 10;
    }
}
```

两种类型,属性全跟 `Type` 走:

| Type | 半径 | 血量 | 分值 | 速度 | 速度来源 |
|---|---|---|---|---|---|
| NORMAL(普通绿僵) | 18 | 1 | 10 | 较快 | `1.0 + random` |
| BRUTE(壮汉) | 26 | 3 | 25 | 较慢 | `0.6 + random*0.4` |

> 壮汉**更肉(3 血,要打 3 下)、更大(半径 26)、分更高(25),但更慢**——典型的「坦克型」敌人。普通僵一刀一个(1 血)、分少(10)、快。

### `update()` —— 追踪玩家 + 朝向 + 闪白衰减

```java
public void update() {
    double dx = targetX - x;                // ① 朝目标的方向向量
    double dy = targetY - y;
    double dist = Math.hypot(dx, dy);
    if (dist > 1e-4) {
        x += dx / dist * speed;             // ② 归一化后按速度移动(永远朝玩家)
        y += dy / dist * speed;
    }
    if (dx < -1e-4) facingLeft = true;       // ③ 据水平方向更新朝向
    else if (dx > 1e-4) facingLeft = false;
    if (hitFlash > 0) hitFlash--;            // ④ 闪白衰减
    animFrame++;                            // ⑤ 动画帧 +1(身体起伏用)
}
```

- **①② 追踪**:`targetX/Y` 是玩家位置(由 `GameController.update` 每帧 `z.setTarget(player.getX/Y)` 回填)。`dx/dist` = 归一化方向(长度 1),乘 `speed` = 这一帧走多远。**所以僵尸永远直线朝玩家追**——不管玩家挪到哪,下一帧方向重算。
- **③ 朝向**:据 `dx` 正负更新 `facingLeft`,`draw` 据此决定要不要水平镜像(和 Player 的镜像同款技巧)。
- **④ 闪白**:`hitFlash` 衰减(被打到时 `takeDamage` 设成 6,6 帧后归零)。
- **⑤ `animFrame`**:`draw` 里用它算身体起伏(`sin(animFrame*0.25)`),让僵尸「蹒跚前行」的摇摆感。

### `draw()` —— 贴图 + 镜像 + 闪白 + 头顶血条

结构和 `Player.draw` 很像(贴精灵图 + `facingLeft` 镜像 + `hitFlash` 闪 + 回退画绿圆),多两样:

**(a) 闪白(不是红闪)**:Zombie 被打时**泛白**(`hitFlash`),用 `AlphaComposite.SRC_ATOP` 染白——和 Player 红闪**同款 SRC_ATOP 技巧**,只是颜色是 `Color.WHITE` 而不是红。区分:玩家受伤 → 红闪(危险感);僵尸被打 → 白闪(命中反馈)。

**(b) 头顶血条**(仅壮汉):

```java
if (maxHp > 1 && hp < maxHp) {              // 多血量 && 已掉血 才显示
    int barW = radius * 2;
    int barY = iy - radius - 10;            // 头顶
    g.setColor(new Color(50, 50, 50));       // 暗底
    g.fillRect(barX, barY, barW, 4);
    g.setColor(new Color(220, 70, 70));      // 红前景,按 hp/maxHp 比例
    g.fillRect(barX, barY, (int)(barW * hp / (double)maxHp), 4);
}
```

- 只有 `maxHp > 1`(壮汉 3 血)才画血条;普通僵 1 血、一打就死,不画。
- 且 `hp < maxHp`(已掉血)才画;满血时不画(干净)。
- 血条 = 暗底 + 红前景按 `hp/maxHp` 比例——经典血条画法。

> 回退分支(精灵图读不到):画绿圆 + 描边 + 红眼 + 嘴(比 Player 的橙猫简单)。和 Player 一样的「读图失败兜底」思想。

### `takeDamage` vs `kill` —— 两种「死」⭐

```java
public void takeDamage(int dmg) {     // 被子弹打:扣血 + 闪白
    hp -= dmg;
    hitFlash = 6;
}
public void kill() {                   // 撞到玩家后消失:hp=0,但不计分!
    hp = 0;
}
public boolean isDead() { return hp <= 0; }
```

**两个「死」的语义**(重要,03 §2.2 也强调过):

| 方法 | 谁调 | hp 怎么变 | 计分? | 场景 |
|---|---|---|---|---|
| `takeDamage(dmg)` | 子弹命中(`handleCollisions`) | 扣血 + 闪白 | 血量归零时**计分**(+10/+25) | 被你打死 |
| `kill()` | 撞到玩家(`handleCollisions`) | 直接 hp=0 | **不计分** | 撞你后消失(防反复扣血) |

> 为什么 `kill` 不计分:僵尸撞到玩家后必须消失(否则一只僵尸能反复撞、把你秒杀),但它是「自己撞死」的不是「被你打死」的,不该给分。所以专门用 `kill()` 区分——**同样是 hp=0,但 takeDamage 路径才计分**。`GameController.handleCollisions` 里两种死都会让 `z.isDead()` 为真而被清理,但只有 takeDamage 打死的那条路才 `score += z.getScoreValue()`。

### 和 Player 的异同(对照)

| | Player | Zombie |
|---|---|---|
| 实现 GameObject | ✅ | ✅ |
| 精灵图 + 懒加载 | ✅ SPRITE | ✅ SPRITE(同款) |
| draw 镜像 | ✅ faceLeft | ✅ facingLeft |
| 闪色 | 受伤红闪(SRC_ATOP 红) | 被打白闪(SRC_ATOP 白) |
| 移动 | 控制器回填位置(不自己动) | **自己追踪**(update 里朝 targetX/Y 走) |
| 血条 | HUD 里画 | **头顶小血条**(仅壮汉) |
| 死法 | 一种(hp 归零 → 游戏结束) | **两种**(takeDamage 计分 / kill 不计分) |
| 类型 | 一种 | **两种**(NORMAL/BRUTE,属性随 Type) |

## 四、`Bullet` —— 子弹

> 文件:[`game/Bullet.java`](../../src/main/java/com/game/game/Bullet.java)
> 谁创建/清理:`GameController.shoot()` 在枪口位置 `new` 出来(`vx/vy` = `cos/sin(angle) × BULLET_SPEED`),丢进 `bullets` 列表;每帧 `update` 飞行 + `draw`,命中僵尸或飞出画布后 `bullets.removeIf(b -> b.isDead() || b.isOffscreen(...))` 清掉。
> 三个 `GameObject` 实现里**最简单**的一个:匀速直线飞,不转弯、不减速。

### 字段一览(逐个注释)

```java
private double x, y;            // 位置
private final double vx, vy;    // 速度向量(像素/帧);final——发射后方向/速度不变
private final int radius = 5;   // 半径(碰撞 + 画圆大小)
private boolean dead;           // 命中标记(打中僵尸后 setDead(true),等清理)
```

> 注意 `vx/vy` 是 `final`:子弹**发射后方向和速度就定了**,不再变(不像 Particle 会阻尼减速、不像 Zombie 会追踪变向)。这就是「匀速直线」。

### `update()` —— 匀速直线(最简)

```java
public void update() {
    x += vx;     // 每帧按速度向量平移
    y += vy;
}
```

就这两行。和另两个对象对比:

- **Bullet**:匀速直线(没阻尼)——子弹该一直飞。
- **Particle**:`x+=vx` 后还有 `vx *= 0.92` 阻尼——火花飞溅后减速。
- **Zombie**:每帧重算方向(朝玩家)——追踪。

三种对象的 `update` 各体现一种**运动模型**。

### `draw()` —— 四层叠出"发光子弹"

```java
public void draw(Graphics2D g) {
    int ix = (int) x; int iy = (int) y;
    // ① 外层光晕:半透明黄大圆
    g.setColor(new Color(255, 220, 80, 70));
    g.fillOval(ix - radius - 3, iy - radius - 3, (radius + 3) * 2, (radius + 3) * 2);
    // ② 拖尾:沿速度反方向的渐淡短线
    g.setColor(new Color(255, 200, 40, 180));
    g.drawLine(ix, iy, (int) (x - vx * 1.6), (int) (y - vy * 1.6));
    // ③ 弹芯:亮黄圆
    g.setColor(Color.YELLOW);
    g.fillOval(ix - radius, iy - radius, radius * 2, radius * 2);
    // ④ 高光点:白色小圆
    g.setColor(Color.WHITE);
    g.fillOval(ix - 2, iy - 2, 4, 4);
}
```

四层从大到小、从淡到亮叠出一个「发光子弹」:

| 层 | 画法 | 颜色 | 作用 |
|---|---|---|---|
| ① 光晕 | `fillOval` 大圆(radius+3) | 半透明黄 alpha=70 | 外圈柔光 |
| ② 拖尾 | `drawLine` 向速度反方向 | 半透明黄 alpha=180 | 运动模糊感 |
| ③ 弹芯 | `fillOval` 圆(radius) | 亮黄 | 子弹本体 |
| ④ 高光 | `fillOval` 小圆(2px) | 白 | 立体高光点 |

**拖尾小技巧**:`drawLine(ix, iy, x - vx*1.6, y - vy*1.6)` —— 从当前位置往**速度反方向**画一段(长度 = 速度的 1.6 倍)。每帧重画、子弹在动,这条短线就形成「尾巴跟着飞」的拖尾。`vx/vy` 是飞行方向,反方向就是尾巴。

> 颜色都用 `new Color(r,g,b,alpha)` 的 **4 参数版**(带透明度)——和 Particle.draw 一样直接用带 alpha 的颜色画半透明,**不用 SRC_ATOP**(子弹是直接画在画布上的圆,没有透明背景要避让)。

### 出界 + 命中:两种"消失"

```java
public boolean isOffscreen(int w, int h) {     // 飞出画布
    return x < -radius || x > w + radius || y < -radius || y > h + radius;
}
public boolean isDead() { return dead; }        // 命中标记
public void setDead(boolean dead) { ... }
```

子弹消失有**两种情况**,都在 `GameController` 清理时一起处理:

```java
bullets.removeIf(b -> b.isDead() || b.isOffscreen(WIDTH, HEIGHT));
```

- `isOffscreen`:飞出画布(打偏了)→ 清掉。
- `isDead`(命中):打中僵尸后 `handleCollisions` 里 `b.setDead(true)` → 清掉。**一颗子弹只命中一只僵尸**(碰撞里有 `break`),命中即标记 dead、当帧清理。

> 对比 Zombie 的「两种死」(takeDamage 计分 / kill 不计分):Bullet 的两种消失都是「清掉」、没计分区别——子弹本身不计分,计分的是「僵尸被打死」(在 Zombie 那边算)。

### 小自测

1. **Bullet 的 `vx/vy` 为什么是 final?和 Particle 的速度有什么不同?**
2. **拖尾那段 `drawLine(ix,iy, x-vx*1.6, y-vy*1.6)` 为什么是「速度反方向」?**
3. **子弹消失有哪两种情况?分别在哪判断?**
4. **一颗子弹能打几只僵尸?为什么?**

<details><summary>答案</summary>

1. final = 发射后方向/速度不变 → 匀速直线。Particle 的速度不是 final(每帧 `vx*=0.92` 阻尼减速);Bullet 不减速、一直飞。
2. 拖尾要画在子弹「后面」,即和飞行方向相反。`vx/vy` 是飞行方向,`-vx*1.6` 就是反方向一段,所以线从当前位置往尾巴方向画。
3. 两种:① 飞出画布(`isOffscreen`,x/y 超出画布)② 命中僵尸(`isDead`,打中后 `setDead(true)`)。都在 `GameController` 的 `bullets.removeIf(b -> b.isDead() || b.isOffscreen(...))` 里清理。
4. **一只**。`handleCollisions` 里命中后 `break` 退出内层循环,不再检测其他僵尸。

</details>
