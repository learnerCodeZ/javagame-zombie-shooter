# 第一站:GameApp(程序入口)

> 文件:[`src/main/java/com/game/GameApp.java`](../../src/main/java/com/game/GameApp.java)(就 25 行,但藏着 Swing 必考点)

## 逐行看

```java
package com.game;

import com.game.ui.LoginFrame;
import com.game.util.UIStyle;

import javax.swing.SwingUtilities;

public class GameApp {

    public static void main(String[] args) {
        UIStyle.initGlobalTheme();
        System.out.println("=== 打僵尸射击游戏 ===");
        System.out.println("阶段③：启动 Swing 登录界面。");
        SwingUtilities.invokeLater(() -> new LoginFrame().setVisible(true));
    }
}
```

### 逐行拆解

**① `package com.game;`** —— 包声明
- Java 用「包」管理类的命名空间(避免类名冲突)。
- **必须和目录结构一一对应**:`com.game` ↔ `src/main/java/com/game/`。
- 必须是文件第一行(注释除外),末尾分号。

**② 三行 `import ...`** —— 引入要用的类
- 要用**不在本包**的类,就得 import,告诉编译器"这类在哪"。
- `com.game.ui.LoginFrame` 的全路径 ↔ 目录 `com/game/ui/LoginFrame.java`。
- `javax.swing.SwingUtilities` 是 **JDK 自带**的(`javax.swing` 包),不用安装。
- 同一个包里的类互相用,**不需要** import。

**③ `public class GameApp {`** —— 定义公开类
- `public`:公开,谁都能访问。
- ⚠️ **文件名必须和 public 类名完全一致** → 所以这个文件叫 `GameApp.java`。
- 一个 `.java` 文件**只能有一个 public 类**。

**④ `public static void main(String[] args)`** —— 程序入口 ⭐
逐个关键字拆(每个都有讲究):
- `public`:JVM 要从外部调用它,必须公开。
- `static`:**静态方法**——不用 `new GameApp()` 建对象就能直接调。JVM 启动那一刻还没有任何实例,所以入口必须 static。
- `void`:无返回值。
- `main`:**方法名写死**——JVM 固定找 `main` 这个名字,写成 `Main` / `start` 都不认。
- `String[] args`:命令行参数。命令行 `java GameApp a b` → `args = ["a","b"]`。本项目没用,但**签名必须长这样 JVM 才认**。

**⑤ `UIStyle.initGlobalTheme();`** —— 设全局暗色主题
- 调 `UIStyle` 的**静态方法**(类名直接调,不用 new)。
- 它内部用 `UIManager.put(...)` 改了 Swing 几十项颜色/字体默认值(详见下方「`UIManager` 与全局主题」)。
- **必须放在创建任何组件之前**,否则已经建好的组件不会跟着变。

**⑥ 两行 `System.out.println(...)`** —— 控制台打印
- `System.out` 是**标准输出流**(默认 = 运行时的控制台黑窗口)。
- 启动提示 / 调试用,**对界面无影响**,删了程序也照跑。

**⑦ `SwingUtilities.invokeLater(() -> new LoginFrame().setVisible(true));`** —— ⭐ 在 EDT 上开登录窗
拆开看:
- `new LoginFrame()` —— 创建登录窗对象(一个 `JFrame`)。
- `.setVisible(true)` —— **让它显示**。⚠️ `JFrame` **默认是隐藏的**,必须 `setVisible(true)` 才看得见——这是新手最常踩的坑("我 new 了窗口怎么没出现?")。
- `() -> ...` —— **lambda**(无参),等价于匿名内部类的简写:
  ```java
  // 等价的老写法(JDK 8 之前只能这么写):
  SwingUtilities.invokeLater(new Runnable() {
      @Override
      public void run() {
          new LoginFrame().setVisible(true);
      }
  });
  ```
- 整行 = 把"建窗 + 显示"这件事**投递给 EDT** 执行(原因见下一节)。

> **最后一句**:`main` 跑完这 7 步就**返回了**,但程序**不会退出**——因为 EDT 还活着、还有可见窗口,JVM 会一直等到所有窗口关闭才结束。所以别看 main 这么短,它"点燃"了整个界面循环。

## `UIManager` 与全局主题(展开第 ⑤ 行 `initGlobalTheme`)

`initGlobalTheme()` 干的事:**改 Swing 全局外观字典 `UIManager` 的默认值**,从而一键把界面换成暗色主题。

### 先分清:`UIManager` 不是我们写的 ⚠️

`UIManager` 是 **JDK 自带的 Swing 类**(`javax.swing.UIManager`,装 Java 就有),**不是项目里的代码**。我们写的 [`UIStyle`](../../src/main/java/com/game/util/UIStyle.java) 只是**导入并调用**它——就像调 `System.out.println`,`System` 也是 JDK 的,你只管用、不定义。

| | 谁定义的 | 在哪 |
|---|---|---|
| `UIManager` | **JDK / Swing**(内置) | `javax.swing` 包,JDK 内部 |
| `UIStyle` | **我们自己写的** | [src/main/java/com/game/util/UIStyle.java](../../src/main/java/com/game/util/UIStyle.java) |

所以下文「改 `UIManager` 默认值」= 调它提供的 `put(...)` 方法,**不是改它的源码**。

> 验证:IntelliJ 里按住 Ctrl 点 `UIManager` → 跳进 JDK 的 class(不在你的项目目录);Ctrl+点 `UIStyle` → 才跳进你项目里的 `UIStyle.java`。

### `UIManager` 是什么

Swing 有个**全局大字典** `UIManager`,存着所有组件的"默认长相",用**字符串键**查:

- `"TextField.background"` → 所有文本框的默认背景
- `"Button.font"` → 所有按钮的默认字体
- `"Table.gridColor"` → 所有表格的网格线颜色

**每个组件被创建/绘制时,都去 `UIManager` 查"我该长啥样"**。所以:在**创建组件之前**改 `UIManager`,后面所有组件自动用新值 → 一键换肤。

```java
UIManager.put("TextField.background", FIELD);   // FIELD = #2c313c
// = 「以后所有 JTextField 默认背景都用这个深灰」
```

`initGlobalTheme()` 一口气 `put` 了几十项,覆盖 Panel / Label / Button / TextField / PasswordField / Table / TableHeader / ScrollPane / ScrollBar / CheckBox / RadioButton / ComboBox 的底色、字色、字体、选中色。

### 默认值被改成了什么(就是 `UIStyle` 顶部那组颜色常量)

| 常量 | 值 | 用途 |
|---|---|---|
| `BG` | `#16181f` | 窗体最深底 |
| `PANEL` | `#232732` | 卡片 / 面板底 |
| `FIELD` | `#2c313c` | 输入框 / 表格行底 |
| `BORDER` | `#3a4150` | 边框 / 网格 |
| `PRIMARY` | `#ff8a1e` | 主色**橙**(主按钮、标题——橙猫) |
| `SECONDARY` | `#3dd578` | 次色**绿**(次按钮——绿僵尸) |
| `DANGER` | `#ef4f4f` | 危险**红**(删除、退出) |
| `TEXT` | `#f1f2f6` | 主文字(近白) |
| `MUTED` | `#9aa1ad` | 次要文字(灰) |
| `SELECTION` | `#8b592e` | 选中色(暗橙) |

### 两个要点

1. **时机**:必须在创建任何组件**之前**调(所以 `GameApp.main` 第一行就调)。晚了的话,已建好的组件不会自动变。
2. **为什么这么干(而不是每个组件手动 setColor)**:**一处改、处处生效**——全站风格统一,改主题只动 `UIStyle` 一个文件。这叫 **DRY**(不重复自己)。

> 方法开头先 `setLookAndFeel(跨平台 Metal L&F)`,因为不同 L&F 对 `UIManager.put` 的尊重程度不同,Metal 最老实采纳这些覆盖值,保证 JOptionPane / JTable 这些"非自定义"组件也稳定变暗。

## 必考点:`invokeLater` 与 EDT ⭐

> 这一节是**整個 Swing 的基石**——后面每一站的按钮回调、Timer、游戏循环都绕不开。讲透它,你看任何 UI 代码都不会懵。

### 什么是 EDT

Swing 用一条专门的工作线程,叫 **EDT(Event Dispatch Thread,事件分派线程)**,来处理**所有**界面相关的事。它内部就是个死循环:不停从一个**事件队列**里取任务、执行。这些"任务"包括:

- **重绘请求**(触发 `paintComponent`)
- **鼠标 / 键盘事件**(移动、点击、按键)
- **按钮点击回调**(`actionPerformed`,即你写的 `e -> {...}`)
- **`javax.swing.Timer` 的 tick**

> 所以你点「登录」按钮 → 产生一个事件 → 排进 EDT 队列 → EDT 取出来 → 跑你的 `doLogin()`。**你的 listener 代码天然就在 EDT 上。**

### 铁律:Swing 组件不是线程安全的

`JTextField` / `JButton` 这些组件**内部没加锁**。要是两条线程同时碰同一个组件(比如 EDT 正在重绘它、另一条线程同时 `setText`),就会出现数据错乱、重绘花屏、偶发 NPE。

为了不给每个组件都加昂贵又复杂的锁,Swing 选了**单线程模型**,立下铁律:

> **所有对 Swing 组件的创建和修改,都必须在 EDT 上做。**
> 只要大家都遵守,组件永远只被一条线程碰,自然不会有并发问题。

### `invokeLater` 是什么

`SwingUtilities.invokeLater(...)` 是 Swing 提供的一个**静态方法**,作用是:**把一段代码安全地交给 EDT 去执行**。它是"非 EDT 线程想碰界面"时的标准入口。

**签名**:

```java
SwingUtilities.invokeLater(Runnable doRun)
```

- 参数是 `Runnable`——一个**函数式接口**(只有一个 `run()` 方法),所以可以直接传 lambda:`() -> { ...代码... }`。
- **它做的事**:把这段代码**塞进 EDT 的事件队列**,然后**立即返回**——既不在当前线程执行它,也不等它执行完。EDT 按队列顺序,轮到它时才跑。
- **为什么叫 "Later"**:你"现在"投递,代码"稍后(later)"才在 EDT 上真正执行——**异步、不阻塞**当前线程。

**最简例子**(在任意线程里,想安全地改界面):

```java
SwingUtilities.invokeLater(() -> {
    myLabel.setText("更新了");   // 这行会在 EDT 上执行,所以安全
});
```

**同胞方法 `invokeAndWait(任务)`**:同样把任务排进 EDT,但会**阻塞当前线程、等任务跑完**才返回。本项目 `main` 用 `later` 就够了(不用等窗口建好)。

### `main` 为什么必须用它

- `main` 由 JVM 在**主线程**上调用,**不是 EDT**。
- 所以在 main 里直接写 `new LoginFrame()` = 在主线程造组件 = **违反铁律**(见上)。
- 于是用 `SwingUtilities.invokeLater(() -> new LoginFrame().setVisible(true))`,把"建登录窗"这件事**转交给 EDT** 执行——合规。

### 怎么判断「这段代码在不在 EDT」⭐(改代码必会)

| 代码位置 | 跑在 EDT 吗? | 想碰界面要 `invokeLater` 吗? |
|---|---|---|
| `main` 方法体 | ❌ 主线程 | ✅ **要** |
| 按钮回调 `e -> {...}` / `actionPerformed` | ✅ | ❌ |
| `MouseListener` / `MouseMotionListener` 回调 | ✅ | ❌ |
| `javax.swing.Timer` 的 tick | ✅ | ❌ |
| 自己 `new Thread(...)` 开的后台线程 | ❌ | ✅ **要** |
| `GameController` 这种纯逻辑类 | **看调用方**:由 Timer tick 调 → 在 EDT | — |

> **判别口诀**:只要代码**不在**「按钮回调 / 监听器 / Timer tick / 已确认在 EDT 的调用链」里,而它又想创建或改界面组件 → 就得拿 `SwingUtilities.invokeLater(...)` 包起来。

### 坑:在 EDT 上干重活会卡界面 ⚠️

EDT 是**唯一**负责画界面的线程。如果你在 EDT 上跑一个慢操作(比如连数据库查几秒),这段时间**界面完全卡死**——不重绘、点按钮没反应、动画停。

- 本项目**真踩了这个坑**(见 [已知问题 #3](../../docs/已知问题与后续优化.md)):DAO 在 EDT 上同步调 JDBC,MySQL 一慢界面就冻。
- 正确做法:用 `SwingWorker`(或线程池)把慢活儿挪到后台线程,完成后 `invokeLater` 回 EDT 更新界面。
- 这也正是 [`SoundUtil`](../../src/main/java/com/game/util/SoundUtil.java) 用**单独的守护线程**播音效的原因——**别占着 EDT**。

## 自测(答案在下方)

> **Q**:把第 23 行改成**直接** `new LoginFrame().setVisible(true);`(去掉 `invokeLater`),程序还能跑吗?会有什么问题?

<details><summary>答案</summary>

**能跑,但不安全**。Swing 规定组件必须在 EDT 上创建,但 JVM 不会强制拦截——所以多半能正常显示窗口,偶尔会出现重绘错乱、偶发 NPE 等"不可预期"的问题。这是典型的"**能跑但错的**"陷阱,答辩时如果被问"为什么用 invokeLater",答:**保证 Swing 组件在 EDT 上创建,避免多线程竞态**。

</details>

# 第二站:登录切片(一条请求穿三层)

> 搞懂这条线,你就掌握了**大半个项目**——注册/改密/忘记密码都是同一个套路。

## 2.0 全景:点「登录」后发生了什么

```
用户在 LoginFrame 输入手机号+密码,点登录
        │  (UI 层)
        ▼
LoginFrame.doLogin()
  取值 → 校验非空 → 校验 11 位数字 → 调 UserDao.login(phone, password)
        │  (跨到数据层)
        ▼
UserDao.login()
  把密码做 MD5 → 拼 PreparedStatement → 调 DBUtil.getConnection() 拿连接
        │  (跨到 DB)
        ▼
MySQL:  SELECT ... FROM user WHERE phone=? AND password=?
        │  返回结果集
        ▼
UserDao 把一行结果「映射」成 User 对象(或 null)
        │
        ▼
LoginFrame: user==null → 提示「手机号或密码错误」
            否则 → dispose() 关登录窗 → new MainFrame(user).setVisible(true)
```

**记住这张图**,下面五个文件就是图里每个框。

## 2.1 LoginFrame.doLogin —— 界面层

> 文件:[`ui/LoginFrame.java` · `doLogin` 方法(第 115–146 行)](../../src/main/java/com/game/ui/LoginFrame.java#L115)
> 📍 关键行:L116-17 取值 · L118 非空校验 · **L123 `\d{11}` 校验** · **L130 调 `login`(跨到数据层)** · L138 `user==null`→失败 · L144-45 `dispose()`+开 `MainFrame`

```java
private void doLogin() {
    String phone = phoneField.getText().trim();          // ① 取值(trim 去首尾空格)
    String password = new String(passwordField.getPassword());
    if (phone.isEmpty() || password.isEmpty()) {         // ② 非空校验
        JOptionPane.showMessageDialog(this, "请输入手机号和密码", ...);
        return;
    }
    if (!phone.matches("\\d{11}")) {                      // ③ 格式校验:11 位数字
        JOptionPane.showMessageDialog(this, "手机号格式不正确...", ...);
        return;
    }
    User user;
    try {
        user = new UserDao().login(phone, password);      // ④ ⭐ 调数据层(核心)
    } catch (Exception ex) {                              //    DB 连不上等异常兜底
        JOptionPane.showMessageDialog(this, "数据库连接失败...", ...);
        return;
    }
    if (user == null) {                                   // ⑤ 登录失败(查不到/密码错)
        JOptionPane.showMessageDialog(this, "手机号或密码错误", ...);
        passwordField.setText("");
        return;
    }
    dispose();                                            // ⑥ 关登录窗
    new MainFrame(user).setVisible(true);                 //    开主菜单,把 user 传过去
}
```

**关键点**:
- 界面层**只管**:取值 → 校验 → 调 DAO → 根据结果弹框/跳窗。**不含任何 SQL**。
- `JPasswordField.getPassword()` 返回 `char[]`(安全考虑,用完可清零),要 `new String(...)` 转成字符串。
- `dispose()` 关当前窗;`new MainFrame(user)` 把登录用户对象**传给主菜单**(后面排行榜/改密都要用 `user.getId()`)。

## 2.2 UserDao.login —— 数据访问层 ⭐

> 文件:[`dao/UserDao.java` · `login` 方法(第 73–99 行)](../../src/main/java/com/game/dao/UserDao.java#L73)
> 📍 关键行:**L74-75 SQL `WHERE phone=? AND password=?`** · L76 `DBUtil.getConnection()` · L78 填手机号 · **L79 密码先 `MD5Util.md5`** · L80 `executeQuery` · L81-93 映射成 User · L98 查不到返 `null`
> (同文件相关:`findByPhone` L26-40、`register` L48-64、`map` L222-234)

```java
public User login(String phone, String password) {
    String sql = "SELECT id, phone, password, nickname, role, create_time FROM user "
               + "WHERE phone = ? AND password = ?";      // ① ? 是占位符
    try (Connection conn = DBUtil.getConnection();        // ② 从 DBUtil 拿连接
         PreparedStatement ps = conn.prepareStatement(sql)) {
        ps.setString(1, phone);                           // ③ 填第 1 个 ?
        ps.setString(2, MD5Util.md5(password));           //   填第 2 个 ?(密码先 MD5)
        try (ResultSet rs = ps.executeQuery()) {          // ④ 执行查询
            if (rs.next()) {                              // ⑤ 有结果 → 映射成对象
                User user = new User();
                user.setId(rs.getInt("id"));
                user.setPhone(rs.getString("phone"));
                ... // 其余字段同理
                return user;
            }
        }
    } catch (SQLException e) { e.printStackTrace(); }
    return null;                                          // ⑥ 查不到 → null
}
```

**必考点 ⭐ `PreparedStatement` 防注入**:
- 为什么不直接拼字符串 `"WHERE phone='" + phone + "'"`?因为用户可以在手机号框输入 `' OR 1=1 --`,拼出来的 SQL 就变成 `WHERE phone='' OR 1=1 --' ...` → **绕过密码校验,查出全部用户**。这叫 **SQL 注入**。
- `PreparedStatement` 用 `?` 占位 + `setString` 填值,驱动会**自动转义**特殊字符,输入 `' OR 1=1` 也只被当成普通字符串匹配 → 攻不破。
- **答辩金句**:「我所有 SQL 都用 PreparedStatement 参数化,杜绝 SQL 注入。」

**`try-with-resources`**(那个 `try (... )`):括号里打开的 `Connection / PreparedStatement / ResultSet` 会在 try 结束时**自动 close**,不用手写 finally。JDK 7+ 语法。

**两个小细节**:
- 密码是**先 MD5 再放进 WHERE**——所以库里存的是哈希,登录时把用户输入也哈希了再比。(等价于 `WHERE phone=? AND password=MD5(用户输入)`。)
- `rs.next()` 移到下一行;初次调用指向第一行。`if` 而非 `while` 因为登录只要一条。

## 2.3 DBUtil —— 连接管理

> 文件:[`dao/DBUtil.java`](../../src/main/java/com/game/dao/DBUtil.java)
> 📍 关键行:**L26-40 `static {}` 块(类加载时读 `db.properties` 一次)** · L29 `getResourceAsStream("db.properties")` · **L47-58 `getConnection()`** · L49 `DriverManager.getConnection` · **L63-82 `close()`(倒序关:rs→stmt→conn)**

三个职责:**读配置 → 给连接 → 关资源**。

**(a) static 静态块读配置**(类加载时执行一次):
```java
private static final String URL, USER, PASSWORD;     // 只读常量
static {
    Properties props = new Properties();
    try (InputStream in = DBUtil.class.getClassLoader().getResourceAsStream("db.properties")) {
        if (in == null) throw new RuntimeException("找不到 db.properties...");
        props.load(in);
    } catch (IOException e) { throw new RuntimeException(...); }
    URL = props.getProperty("url");
    USER = props.getProperty("user");
    PASSWORD = props.getProperty("password");
}
```
- `static {}` 块在**类第一次被用到时执行一次**——配置只读一次,之后 `URL/USER/PASSWORD` 是常量。
- `db.properties` 放在 `src/main/resources/`,**编译后进 classpath**;`getResourceAsStream` 从 classpath 读,不依赖工作目录。
- ⚠️ `db.properties` 含你的 MySQL 密码,**已被 .gitignore 排除**,公开仓库只有 `db.properties.example`。

**(b) getConnection**:`DriverManager.getConnection(URL, USER, PASSWORD)` —— JDBC 标准 API,返回一个连接。失败时抛带友好提示的 RuntimeException(不抛裸 SQLException)。

**(c) close(顺序:ResultSet → Statement → Connection)**:
```java
public static void close(Connection conn, Statement stmt, ResultSet rs) {
    if (rs != null) try { rs.close(); } catch (SQLException ignored) {}
    if (stmt != null) try { stmt.close(); } catch (SQLException ignored) {}
    if (conn != null) try { conn.close(); } catch (SQLException ignored) {}
}
```
- **倒序关**:先开的后关(像栈)。先关结果集,再关语句,最后关连接。
- 每个单独 `try/catch`:一个关失败不能影响后面资源的关闭,否则泄漏。
- 每个 `catch (ignored)`:关闭时的异常没法处理,吞掉即可(连接已经要用完了)。
- > 注:本项目 DAO 现多用 `try-with-resources` 自动关,这个 `close` 主要是给早期/手动管理时用。

## 2.4 MD5Util —— 密码哈希

> 文件:[`util/MD5Util.java` · `md5` 方法(第 26–47 行)](../../src/main/java/com/game/util/MD5Util.java#L26)
> 📍 关键行:L31 `MessageDigest.getInstance("MD5")` · **L32 `digest(...)` 算 16 字节摘要** · L33-41 每字节转 2 位十六进制 · L36 `b & 0xff`(byte 当无符号) · L42 返回 32 位串

```java
public static String md5(String plain) {
    MessageDigest md = MessageDigest.getInstance("MD5");     // 拿到 MD5 摘要器
    byte[] bytes = md.digest(plain.getBytes(UTF_8));         // 算出 16 字节摘要
    StringBuilder sb = new StringBuilder();
    for (byte b : bytes) {
        String hex = Integer.toHexString(b & 0xff);          // 每字节 → 2 位十六进制
        if (hex.length() == 1) sb.append('0');               // 补前导 0(如 0xf → "0f")
        sb.append(hex);
    }
    return sb.toString();                                    // 32 位十六进制串
}
```
- MD5 把任意输入算成**固定 16 字节 = 32 位十六进制**。`"123456"` → `e10adc3949ba59abbe56e057f20f883e`。
- `b & 0xff`:byte 在 Java 是有符号的(-128~127),`& 0xff` 把它当无符号处理(0~255),否则负数 `toHexString` 会出一串 f。
- **为什么存哈希不存明文**:数据库泄露时,明文密码直接暴露;哈希不可逆,攻击者拿到的只是一串摘要。(局限见总纲 §7:MD5 无盐,生产应换 BCrypt。)
