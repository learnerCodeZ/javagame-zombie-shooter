## 一、一句话 + 比喻

**Workflow（工作流）= 一个"指挥多个 AI 小助手分工协作、还能互相检查"的脚本。**

打比方 🏗️：

| 现实 | 对应 |
|---|---|
| 你（包工头） | 我（主 AI） |
| 写一份"施工方案"（谁干什么、什么顺序、怎么验收） | **Workflow 脚本**（一段 JS） |
| 手下的工人（小助手） | **subagent（子智能体）** |
| 工人各自领活干、互不干扰 | `parallel`（并行） |
| 一道工序完了才能进下一道 | `phase`（阶段） |
| 质检员挑刺 | **对抗式验证（adversarial verify）** |

平时我是**一个人从头干到尾**；Workflow 让我能**派多个小助手并行干、再互相审查**，适合"稍大、需要严谨"的活。

---

## 二、Workflow 到底是什么

它是一个**确定性的编排脚本**（JavaScript 写的），你（或我）在里面规定：

- **派几个小助手**、**各自干什么**；
- **什么顺序**（先后 / 并行 / 流水线）；
- **结果怎么传给下一步**；
- **要不要验证**（写完让另一个独立小助手挑刺）。

> 关键区别：平时我（单个 Agent）是"边想边做"；Workflow 是"**先把流程写死成脚本，再让小助手按脚本执行**"——所以叫"确定性"编排。

脚本长这样（骨架）：
```js
export const meta = {            // 必须：名字、说明、阶段
  name: 'build-ui',
  description: '构建并验证界面',
  phases: [{ title: 'Implement' }, { title: 'Verify' }, { title: 'Fix' }],
}

phase('Implement')
const code = await agent('写登录界面...')     // 派一个助手写代码

phase('Verify')
const checks = await parallel([               // 同时派多个助手检查
  () => agent('编译检查...'),
  () => agent('代码审查...'),
])
```

## 三、为什么要用 / 什么时候用

三大目的（来自工具设计）：

| 目的 | 通俗解释 | 典型场景 |
|---|---|---|
| **更全面** | 把大活拆成小块，多个助手分头覆盖 | 大范围审查、迁移 |
| **更可靠** | 写完后让独立助手"挑刺"再定稿，减少错漏 | 代码审查、找 bug |
| **更大规模** | 一个上下文装不下的量，靠多个助手分担 | 几百个文件的处理 |

**常见工作流类型**：理解（读懂代码）→ 设计（出多方案评分）→ 审查（找 bug + 验证）→ 研究（多源检索）→ 迁移（批量改）。

> ⚠️ 不是所有活都该用 Workflow。**一句话能答的、改个错别字这种小活，直接做就行**。Workflow 是为"值得拆分 + 验证"的稍大任务准备的。

## 四、核心积木（构建块）

| 积木 | 作用 | 像什么 |
|---|---|---|
| `agent(提示, 选项)` | **派一个小助手**干活，返回它的结果 | 派工人 |
| `parallel([任务1, 任务2])` | **同时跑**多个任务，等全部完成（有"栅栏"） | 几个工人同时开工，干完一起汇报 |
| `pipeline(物品, 工序1, 工序2)` | 每个物品**流水线**穿过各工序，无栅栏 | 流水线：A 在工序3 时 B 还在工序1 |
| `phase('标题')` | 把小助手**分组**显示进度 | 施工阶段标签 |
| `schema`（选项） | 强制小助手返回**结构化数据**（JSON） | 让工人按表格填，别自由发挥 |
| `log('消息')` | 打一条**进度**给你看 | 工地广播 |

**关于 `parallel` vs `pipeline`（最容易混）**：
- `parallel` = **齐头并进，等最慢的那个**（需要全部结果一起用时用）。
- `pipeline` = **流水线，不等齐**（每个物品独立走完全流程，省时间）。

## 五、几种"质量套路"（高级模式，了解即可）

| 套路 | 干什么 |
|---|---|
| **对抗式验证** | 派几个"杠精"助手专门反驳结论，多数反驳就否决 → 防止"看起来对其实错" |
| **评审团** | 出 N 个方案，评委打分，取最优 + 融合其他亮点 |
| **多角度扫描** | 每个助手用不同方式搜，互不干扰，覆盖更全 |
| **循环到没新货** | 一直派助手找，连续几轮没新发现才停 |
| **完整性批评家** | 最后派一个助手问"还漏了啥"，漏的变成下一轮任务 |

> 阶段③我用的是最朴素的"**实现 → 验证 → 修复**"，已经能抓到真实 bug 了。

## 六、我在阶段③怎么用的（具体例子）

```
阶段③ Workflow 流程：
┌─────────────┐
│ Implement    │  1 个助手：按详细规格写 LoginFrame/RegisterFrame/MainFrame/GameApp
└──────┬──────┘
       ▼
┌─────────────┐  并行 2 个助手：
│ Verify       │   ① javac 编译检查
│ (parallel)   │   ② 代码审查（跳转流/DAO调用/校验/规范）
└──────┬──────┘
       ▼（审查发现 1 个中等 bug）
┌─────────────┐
│ Fix          │  1 个助手：修 bug + 重新编译验证通过
└─────────────┘
```

**抓到的 bug**：登录窗跳注册窗时用了 `setVisible(false)`（只隐藏不释放），反复操作会**窗口泄漏**。审查助手独立看出了这个我自己写也容易漏的问题——这就是 Workflow"写完即审查"的价值。

实际开销：4 个小助手、约 10 万 token、十几分钟（后台跑，你能用 `/workflows` 看进度）。

## 七、怎么用（你的视角）

**你一般不需要自己写 Workflow 脚本**。两种触发方式：

1. **你主动要**：跟我说"**用工作流来做 X**"或"**用多个助手并行**"，我就编排一个。
2. **自动触发**：开了 **ultracode 模式**（追求最彻底、不省 token）时，我会默认对大任务用 Workflow。

启动后：
- 它在**后台**跑，我会收到完成通知再汇报。
- 你想看进度：输入框敲 **`/workflows`**，能看到实时的阶段树和每个助手的状态。
- 跑完我复核结果 → 给你汇报 → 写复盘。

## 八、Workflow vs 单个 Agent（区别）

| 维度 | 单个 Agent（平时） | Workflow |
|---|---|---|
| 执行者 | 我一个人 | 多个小助手 |
| 控制方式 | 我边想边决定下一步 | 脚本提前写死流程 |
| 并行 | 没有（顺序做） | 有（`parallel`/`pipeline`） |
| 自我审查 | 我自己复查 | 独立助手挑刺（更客观） |
| 适合 | 日常对话、小改、单线任务 | 大任务、要严谨、要并行 |
| 开销 | 小 | 大（多个助手 = 更多 token/时间） |

## 九、几个关键细节（别踩坑）

1. **小助手的"返回值"就是它的最后一段文字**——它是给我（脚本）看的"数据"，不是给你看的消息。所以小助手要"说人话给数据"，要结构化就用 `schema`。
2. **脚本里没有 `Date.now()` / `Math.random()` / `new Date()`**（会破坏断点续跑），要随机就靠"每个助手提示不同"。
3. **并发有上限**（大约 `CPU核数-2`，最多 16 个同时），超出就排队。
4. **后台跑**：启动后立刻返回，完成才通知我；不是阻塞等待。
5. **可续跑/可改**：脚本会存成文件，改完用 `resumeFromRunId` 续跑，没改的步骤走缓存。

## 十、一句话总结

> **Workflow 是一段"指挥多个 AI 小助手按既定流程干活、并互相验证"的脚本；适合稍大、需要严谨和并行的任务。你只要说"用工作流来做"，我就编排；想看进度敲 `/workflows`。**

阶段③用它抓到了窗口泄漏 bug——这就是它最实在的价值：**写完立刻被独立审查，少踩坑**。
