| 状态 | ✅ 完成、全量编译通过、`TestGameLogic` 运行验证、4 维度对抗 review 0 问题 |
| 日期 | 2026-07-10 |

---

## 一、做了什么

点"开始游戏" → 弹 `JOptionPane` 选 **简单 / 困难**（默认高亮"简单"，点 X 可取消）→ 带所选难度进入 `GameWindow` → `GameController` **按难度注入**刷怪与伤害参数。游戏窗口标题、左上 HUD 都显示当前难度；**"再来一局"保持同一难度、不掉档**。

玩家能力（移速 / 血量 / 子弹）两档**完全一致**——难度只来自威胁变强，不靠削弱玩家。

---

## 二、难度数值

| 参数 | 简单（= 改动前原值） | 困难 | 效果差异 |
|---|---|---|---|
| 初始刷怪间隔 | 90 帧（1.5s/只） | 60 帧（1.0s/只） | 开局更紧 |
| 封顶刷怪间隔 | 30 帧（0.5s/只） | 20 帧（0.33s/只） | 封顶更密（约 3 只/秒） |
| 壮汉登场秒数 | 10s | 5s | 更早掺肉 |
| 壮汉概率 | 18% | 35% | 占比近翻倍 |
| 撞击伤害 | 20 | 25 | 容错 5 下→4 下 |

> 两档都用**同一个公式** `间隔 = max(封顶, 初始 - 存活秒数)`、同一个壮汉判定 `存活秒 ≥ 门槛 && 随机 < 概率`——只是代入的数不同。困难约 **40 秒**压到封顶（简单要 60 秒）。

## 三、核心知识点 ⭐

### 3.1 用枚举封装"随难度而变"的参数
新建 [`game/Difficulty.java`](../../src/main/java/com/game/game/Difficulty.java)：一个枚举，每个档位自带 6 个字段（`spawnInterval / minSpawn / bruteMinSec / bruteChance / hitDamage / label`）+ 构造方法。调难度只动这一处数值，游戏逻辑里**再无任何写死的魔法数**。
- 要点：**枚举也能有字段和构造方法**——它不只是常量集合，本质是一个"实例数有限、在定义里写死"的类。这里正是它的理想用法（档位有限、每档一组固定参数）。

### 3.2 `static final` 常量 → 实例字段（参数化）⭐ 本扩展最大的设计点
改动前，刷怪间隔等是 `private static final int SPAWN_INTERVAL = 90;`——**全局写死、所有局都一样**。要做难度，它们必须**能按局变化**，于是改成实例字段 `private final Difficulty difficulty;`，构造时按难度赋值，运行时读 `difficulty.spawnInterval` 等。
- 这是**"同一套代码、注入不同参数 → 不同行为"**的思想（构造方法注入，本质是依赖注入的最朴素形式）。
- 没法继续用 `static final`：静态字段属于类、全游戏共享一份，放不下"这局简单、那局困难"。实例字段属于单个 `GameController`（= 一局），每局各持一份难度，天然互不干扰。

### 3.3 难度数据流（端到端） + "再来一局"保持难度
```
MainFrame 开始按钮 → 弹窗选 difficulty → new GameWindow(user, difficulty)
                                          → new GameController(difficulty)
                                          → update() 读 difficulty.* 跑逻辑
```
- `GameWindow` 把难度存为 `private final Difficulty difficulty;` 字段，**两处"再来一局"**（死亡结算 `settle()`、设置菜单的"再来一局"）都 `new GameWindow(currentUser, difficulty)` 原样回传——所以连开几局困难不会偷偷变回简单。
- "退出游戏→回主菜单→再开始"会**重新弹窗**让你选（符合直觉：回了大厅就重新挑）。

### 3.4 无参构造委托默认值（兼容老测试）
保留 `GameController()`，方法体就一句 `this(Difficulty.EASY);`——委托给带参构造、默认简单档。这样老的 [`TestGameLogic`](../../src/main/java/com/game/TestGameLogic.java)（调 `new GameController()`）**无需改动**继续能跑。
- 要点：**构造方法之间可以用 `this(...)` 互相调用**；把"默认值"集中在一个构造里、其它构造委托过去，避免重复初始化代码。

### 3.5 `JOptionPane.showOptionDialog` 弹窗交互
```java
int choice = JOptionPane.showOptionDialog(this, "选择游戏难度", "难度选择",
        DEFAULT_OPTION, QUESTION_MESSAGE, null,
        new Object[]{"简单", "困难"}, "简单");   // 选项数组 + 默认高亮"简单"
if (choice == CLOSED_OPTION) return;             // 点 X 关闭 = 取消，不开始
Difficulty diff = (choice == 1) ? Difficulty.HARD : Difficulty.EASY;
```
- 返回值是**所点选项在数组里的下标**（0=简单，1=困难）；点 X 关闭返回 `CLOSED_OPTION`(-1)。
- 取消时直接 `return`，**不 `dispose()` 主菜单**——用户回到主菜单而非黑屏。

### 3.6 当前难度的视觉反馈（标题 + HUD）
- 窗口标题：`打僵尸 [困难]`。
- HUD：分数行右侧画 `[困难]`，**困难红、简单绿**——一眼知道当前档位，颜色还暗示压力等级。

### 3.7 数值设计哲学：只放大威胁、不削弱玩家
困难**不动**移速/血量/子弹。原因：难度应来自"怪更强"，而非"玩家更弱"。两档玩家手里的工具一模一样，比拼纯粹是操作与压力的对抗——这样困难"难而公平"，不会显得不讲理。

## 四、踩坑 / 收获

- **`{@value #常量}` 在常量变实例字段后失效**：`spawnZombie()` 的 javadoc 原本写 `存活满 {@value #BRUTE_MIN_SEC} 秒`，引用的是静态常量；改成实例字段后 `{@value}` 找不到目标（编译不报错，但文档生成会断）。改写成 `{@link Difficulty#bruteMinSec}` 指向枚举字段。
- **两处"再来一局"是相同文本**：`new GameWindow(currentUser).setVisible(true);` 在 `settle()` 和设置菜单里各出现一次。用 `replace_all` 一次性都改成带 `difficulty`——**漏改任何一处，那一处就会掉档（编译还会直接报错，因为构造签名变了）**。
- **编译器抓得到代码引用、抓不到 javadoc `{@value}`**：删常量后 `javac` 能查出所有代码处的残留引用（编译失败），但 javadoc 里的 `{@value}` 只在生成文档时才暴露。收尾用 `grep` 扫一遍五个常量名兜底，确认零残留。
- **`agents_empty_result` 不等于"没审查"**：对抗式 review 四个 agent 都返回空 findings，但用法统计有 36 次工具调用（≈每个 agent 读 5 文件 + 几次 grep）——说明是真审完没发现问题，而非空跑。判断 review 是否"真做了"要看工具调用量，不能只看返回空。
- **无参构造保留 = 不破坏测试**：删掉 `GameController()` 会让 `TestGameLogic` 编译失败；用 `this(EASY)` 委托，老测试零改动。
