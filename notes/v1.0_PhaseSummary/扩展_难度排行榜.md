## 一、做了什么

难度选择功能上线后，简单/困难战绩混在一个榜里不公平。本次把排行榜按难度拆开：每局战绩记录它是哪个难度（`game_record` 加 `difficulty` 列，老数据自动落 `EASY`），DAO 按难度过滤，排行榜顶栏新增「简单榜 / 困难榜」切换，与原「全局榜 / 我的记录」叠加成 **2 难度 × 2 范围 = 4 种视图**。

---

## 二、核心知识点 ⭐

### 2.1 加列 + 幂等迁移（无损、可重复执行）
新增 [`sql/migrate_v1.2_leaderboard_difficulty.sql`](../../sql/migrate_v1.2_leaderboard_difficulty.sql)，照 `migrate_v1.1` 的风格：
```sql
SET @col := (SELECT COUNT(*) FROM information_schema.columns
             WHERE table_schema='game_db' AND table_name='game_record' AND column_name='difficulty');
SET @ddl := IF(@col = 0,
   'ALTER TABLE game_record ADD COLUMN difficulty VARCHAR(8) NOT NULL DEFAULT ''EASY'' COMMENT ''难度 EASY/HARD''',
   'DO 0');
PREPARE stmt FROM @ddl; EXECUTE stmt; DEALLOCATE PREPARE stmt;
```
- **幂等三件套**：`information_schema` 查列在不在 → `IF(@col=0,'ALTER...','DO 0')` 拼出"加列或空操作" → `PREPARE/EXECUTE/DEALLOCATE` 执行动态 SQL。**跑多少次都不报错**。
- **老数据靠 `DEFAULT 'EASY'` 自动填充**，不需要单独 `UPDATE`——老游戏没难度概念，数值等同简单档，EASY 是唯一合理默认。
- SQL 字符串里的单引号要 `''` 转义（`DEFAULT ''EASY''`）。
- 加索引同理：查 `information_schema.statistics` → 幂等 `ADD INDEX`（见 2.6）。

### 2.2 DAO 按难度过滤（`PreparedStatement` 参数下标）
[`RecordDao`](../../src/main/java/com/game/dao/RecordDao.java) 三处：
- **saveRecord**：INSERT 加 difficulty 列（5 列 5 个 `?`），`ps.setString(5, record.getDifficulty())`。
- **topN(n, difficulty)**：`WHERE g.difficulty = ? ORDER BY g.score DESC LIMIT ?`，注意**参数下标**——difficulty 是第 1 个（`setString`）、n 是第 2 个（`setInt`）。
- **mine(userId, difficulty)**：`WHERE user_id = ? AND difficulty = ?`，userId 第 1、difficulty 第 2。

> 要点：`PreparedStatement` 的 `?` 按**填值顺序**对应下标 1/2/…，改 SQL 加条件时务必同步调 `setX` 的下标，错位会类型不符或查错列。

### 2.3 `map()` 列对等陷阱（避老坑）
`map(rs)` 里新加了 `r.setDifficulty(rs.getString("difficulty"))`。**前提**：`topN` 和 `mine` 的 SELECT 列表都补了 `difficulty` 列。
- 这个项目里 `map`（mine 用）和 `mapWithUser`（topN 用，多出 nickname）是**分开的**，注释明说"避免 mine 因结果集无 nickname 列而报错"。所以凡是 `map` 读的列，**两个 SELECT 都得查出来**，否则运行时 `Column 'xxx' not found`。加 difficulty 时盯住了这点。

### 2.4 改方法签名 → 同步所有调用方
`topN` / `mine` 各加一个 `difficulty` 参数，调用方必须全改：
- [`TestDao`](../../src/main/java/com/game/TestDao.java)：`topN(10,"EASY")`、`mine(id,"EASY")`。
- [`LeaderboardFrame`](../../src/main/java/com/game/ui/LeaderboardFrame.java)：`reload()` 里 `topN(50, currentDifficulty.name())` / `mine(id, currentDifficulty.name())`。
- 漏掉任何一处，**编译器直接报错**——这是静态强类型的好处，签名变更不会静默出错。改前用 grep 摸清所有调用方（`new GameRecord()`/`saveRecord`/`topN`/`mine`），心里有数再动。

### 2.5 难度值往返（大小写一致性）⭐ 隐蔽 bug 高发区
难度字符串在三层之间流转，**大小写必须完全一致**，否则困难记录在困难榜里查不出来（静默返回空）：
- **存**：`GameWindow` → `record.setDifficulty(difficulty.name())` = `"EASY"` / `"HARD"`（大写，Java 枚举 `name()`）。
- **查**：`LeaderboardFrame` → `currentDifficulty.name()`（大写）传给 DAO 的 `WHERE difficulty = ?`。
- **DB 列**：`VARCHAR(8) DEFAULT 'EASY'`（大写）。

三者都是大写、都用 `Difficulty.name()`，天然对齐、零转换。对抗 review 专门查了这一维，确认干净。**这类"存小写、查大写"的不一致是最危险的——不报错、只是查不到，得靠审查主动盯。**

### 2.6 复合索引 + 迁移/全新装结构对齐
`WHERE difficulty=? ORDER BY score DESC` 这个查询，复合索引 `idx_diff_score (difficulty, score)` 最合适（先按 difficulty 定位、再按 score 排好序，免 filesort）。
- 进了 `schema.sql`（全新装）和**迁移**（升级装）两处——对抗 review 发现我一开始只在 schema.sql 加了索引、漏在迁移里，导致"升级库 vs 全新库"结构不一致。补了一段幂等 `ADD INDEX`（查 `information_schema.statistics`）后两者对齐。
- 要点：**迁移和 schema.sql 要产出同一张表**——加列/加索引任一处漏了，两种安装路径就分叉了。

### 2.7 排行榜 UI 拆分（单帧 + 双维度切换）
[`LeaderboardFrame`](../../src/main/java/com/game/ui/LeaderboardFrame.java) 重写：两个字段 `currentDifficulty`（默认 EASY）、`mineMode`（默认 false），一个 `reload()` 按 (难度, 范围) 查询刷新，顶栏 5 个按钮各改一个状态后调 `reload()`，外加一个 `viewLabel` 显示当前视图（文字 + 颜色随难度：简单绿/困难红）。
- **为什么不拆成两个独立窗口/两个主菜单入口**：主菜单只有一个"排行榜"按钮，帧内切换最自然、改动最小；双维度叠加成 4 视图比硬拆两个窗灵活。

## 三、踩坑 / 收获

- **图省事漏了迁移里的索引**：复合索引一开始只进 schema.sql、没进迁移（想着"数据量小省段样板"）。对抗 review 抓到"升级库与全新库结构不一致 + 少了查询优化"，补了幂等 `ADD INDEX`。教训：**迁移和建表脚本要对齐，别只顾一边。**
- **`map()` 读取的列，所有 SELECT 都得有**：项目里 map/mine 无 JOIN、mapWithUser/topN 有 JOIN，是分开的。给 map 加读 `difficulty`，就得保证 mine 的 SELECT 也查这列，否则运行时炸。
- **签名变更靠编译器兜底**：DAO 方法加参数，调用方漏改 → 编译失败，立刻暴露。比"改了逻辑但忘改调用方、运行时才出错"安全得多。
- **难度大小写是隐蔽雷区**：存 `name()`(大写)、查 `name()`、DB 列大写——三处一致才不出错；这种 bug 不报错只查不到，得主动审查。
- **`agents_empty_result` 不等于没审**：本次 4 维度 review，3 个返回空(0 发现)、1 个返回 1 条(low)。看工具调用量(48 次)和"哪一维返回了发现"判断是真审完，而非空跑。
