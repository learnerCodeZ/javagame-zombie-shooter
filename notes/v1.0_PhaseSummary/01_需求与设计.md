## 1. 开发流程（这阶段怎么走过来的）

```
确认选题 → 定技术栈 → 探测环境 → 写计划书 → 写建表脚本 → 扩写成三份正式文档 → 更新计划书
```

1. **确认选题**：从一张 Scratch 射击游戏截图（猫打僵尸）出发，定下"Java 2D 射击小游戏"。
2. **定技术栈**：Java Swing（界面）+ MySQL（数据库）+ JDBC（连接）。
3. **探测环境**：确认本机 `mysql` 客户端在 `D:\MySQL80\bin\`、3306 端口在监听、装了 DBeaver。
4. **写计划书**：先有总纲，明确范围和阶段。
5. **写建表脚本**：把数据库设计直接落地成可执行的 SQL（含测试数据）。
6. **扩写成三份正式文档**：把计划书里的设计概要，拆成需求/设计/数据库三份独立文档，并用 Mermaid 画正式图。
7. **更新计划书**：勾选已完成的交付物，保持文档同步。

> **复盘要点**：先出"粗"的计划书（快速对齐方向），再出"细"的三份文档（正式交付）。这是"**先发散后收敛**"的工程节奏，避免一上来就钻细节。

## 2. 关键设计决策（为什么这么定）

| 决策 | 选择 | 为什么 |
|---|---|---|
| 选题 | 打僵尸射击游戏 | 玩法简单，能把 OOP（玩家/僵尸/子弹）、游戏循环、碰撞都体现出来；又带登录+排行榜，覆盖数据库考点 |
| 界面 | Java Swing | JDK 自带、课设标配、不用额外依赖 |
| 数据库 | MySQL 8 | 本机已装、免费、JDBC 资料多 |
| 连接方式 | JDBC（非 ORM） | 课设要求掌握底层 SQL 与增删改查，JDBC 最直白 |
| 密码存储 | MD5 | 不存明文，安全；登录时同样 MD5 后比对 |
| 图表 | Mermaid | 纯文本可版本控制、VS Code 直接渲染、可导出 PNG 贴答辩 PPT |
| 架构 | 三层（UI/游戏/数据） | 降低耦合，改一层不影响别的 |

## 3. 核心知识点（复盘重点，建议逐条吃透）

### 3.1 需求分析：搞清"做什么"
- **功能需求 vs 非功能需求**：功能需求 = 系统要做什么（登录、射击、存档）；非功能需求 = 做得多好（帧率、健壮性、安全）。两者都要写。
- **模块化编号**（F1.1、F3.4…）：每个功能点有编号，后面设计/测试都能追溯，不会漏。
- **用例（Use Case）**：参与者 + 前置条件 + 主流程 + 异常流。这是软件工程的标配写法，答辩常问。
- **用例图**：画出"谁（参与者）能用哪些功能（用例）"，一眼看清系统边界。

### 3.2 软件架构：为什么要分层
- 三层：**UI 层（窗体）/ 游戏层（玩法逻辑）/ 数据层（JDBC）**。
- **分层的好处**：界面改动不影响数据库逻辑，数据库换实现也不动界面 → **低耦合、好维护**。
- **单向依赖**：上层调下层（UI→游戏→数据），不反向依赖，避免循环依赖。

### 3.3 面向对象：怎么体现 OOP
- **接口抽象**：定义 `GameObject` 接口（`update()` + `draw()`），`Player/Zombie/Bullet` 都实现它。
  - 好处：`GamePanel` 可以用**统一的方式**管理所有对象（一个列表、统一调 `update/draw`）——这就是**多态**的价值。
- **封装**：字段私有 + getter/setter，外部不能乱改内部状态。
- **一对多在代码里**：`User` 拥有多条 `GameRecord`，和数据库的外键关系对应。

### 3.4 数据库设计：从概念到落地
- **设计顺序**：先画 **ER 图**（概念设计，理清实体和关系）→ 再定**表结构**（逻辑设计）→ 最后写**建表 SQL**（物理实现）。
- **一对多**：一个用户多条战绩 → 在"多"的一方（game_record）加外键 `user_id` 指向 user。
- **外键**：保证数据完整性（战绩必须属于一个真实存在的用户）。
- **索引**：`idx_score` 加速排行榜按分数排序；`idx_user` 加速"我的记录"查询。
- **字符集 utf8mb4**：支持中文和 emoji，**防乱码**（比 utf8 更全）。
- **存储引擎 InnoDB**：支持外键和事务（MyISAM 不支持）。
- **删表顺序**：先删有外键的子表（game_record），再删父表（user）——否则报错。

### 3.5 文档与建模：Mermaid
- 四种图各管一摊：
  - `classDiagram` → 类与类关系（设计）
  - `flowchart` → 流程/结构（流程）
  - `erDiagram` → 实体关系（数据库）
  - `sequenceDiagram` → 时序（交互过程）
- **代码即图**：图就是文本，改图=改文本，可版本控制、可 diff——比 Visio 手画更适合程序员。

## 4. 踩坑 / 注意点（记下来，后面阶段别再踩）

| 坑 | 说明 | 关联阶段 |
|---|---|---|
| **跨文档一致性** | 数据库文档的每个字段必须和 `schema.sql` 严格对齐（这次专门核对过） | 本阶段 |
| **Mermaid 语法** | 节点文本含 `/()？<>` 等特殊字符要加引号；泛型用 `~`（如 `List~Zombie~`） | 本阶段 |
| **MySQL 8 时区** | 连接串要带 `serverTimezone=Asia/Shanghai`，否则报错 | 阶段② |
| **中文乱码** | URL 带 `characterEncoding=utf8` + 库用 utf8mb4 | 阶段② |
| **密码比对** | 登录时密码也要先 MD5 再查，不能拿明文比对；且要和 `schema.sql` 测试数据的 MD5 一致 | 阶段② |
| **外键删表顺序** | 先子表后父表 | 本阶段（脚本）/ 阶段② |

## 6. 收获与下一阶段衔接

**本阶段最大的收获**：
- 真正体会了"**先设计后编码**"的价值——表结构、类关系、流程都画清楚后，后面写代码基本是"翻译"文档，心里有底、少返工。
- 学会了用 **Mermaid** 把脑中的设计变成可渲染、可交付的图。
- 把"需求分析—系统设计—数据库设计"这条主线走了一遍，对软件工程流程有了完整认识。

**阶段②（数据层）衔接**：
- 本阶段定好的 **`schema.sql` + 数据库文档**，就是阶段② JDBC 代码要操作的表。
- 阶段②会把"Java ↔ MySQL"打通，先实现 `MD5Util → DBUtil → User/GameRecord → UserDao/RecordDao`，跑通**注册/登录/存战绩/查排行榜**。
- 阶段②会用到 **root 密码**（连库测试），并验证本阶段埋的几个坑（时区、乱码、MD5）。
