```
启动 Workflow → [Implement 写9文件] → [Verify 并行4路: 编译/逻辑审查/集成审查/跑逻辑测试]
             → [Fix 修2个中等bug] → 我独立复核(读核心文件+自编译+重跑测试) → 修hp负数 → 总结
```

**Workflow 这次抓到的真实 bug（2 个中等）**：
1. **结算弹窗会被吞**：`saveRecord` 没包 try/catch，连不上库时抛 `RuntimeException`，后面的结算弹窗被跳过 → 玩家死后卡在终局画面。**修复**：套 try/catch + 失败提示。
2. **X 关窗进程空挂**：进游戏后主菜单已 dispose，GameWindow 是 `DISPOSE_ON_CLOSE`，X 关窗后无可见窗也无 EXIT，JVM 悬挂。**修复**：`windowClosing` 里回主菜单。

**我额外修的**：玩家血量能扣成负数（HUD 显示"血量:-15"）→ `takeDamage` 里 clamp 到 0（本来计划书第九节示例就是 clamp 的）。

> **复盘要点**：游戏 bug 多是"运行时才暴露"的（异常吞掉、进程悬挂、负数血量），光靠编译抓不到。这次"**并行审查 + 无界面逻辑测试**"组合拳正好补上——逻辑测试跑 600 帧直接证明主循环不崩，审查读代码抓到集成层的坑。

## 2. 关键设计决策

| 决策 | 选择 | 为什么 |
|---|---|---|
| 玩家是否移动 | **固定中央**，只瞄准射击 | 贴合 Scratch 原型（猫站着打），降低复杂度；移动留到阶段⑥ |
| 主循环 | `javax.swing.Timer(16ms)` 在 EDT | Swing 自带、EDT 单线程**无并发问题**；约 60 FPS |
| 逻辑与视图分离 | `GameController`(逻辑) + `GamePanel`(渲染/输入) | 不含 Timer 的控制器可**无界面测试**；职责清晰 |
| 对象统一管理 | `GameObject` 接口（`update`+`draw`） | 三类对象多态统一处理，加新对象易扩展 |
| 碰撞模型 | 圆形距离判定（`Math.hypot`） | 简单高效，够用 |
| 删除方式 | `removeIf` 批量删 | 避免**遍历时修改集合**的 `ConcurrentModificationException` |
| 结算通知 | 回调 `onGameOver`（观察者雏形） | 控制器不认识窗口，解耦 |

## 3. 核心知识点（游戏开发入门，复盘重点）⭐

### 3.1 游戏主循环（Game Loop）—— 游戏的心跳
所有实时游戏的本质就是一个**死循环**，每圈叫一"帧"：
```
每帧（约 16ms）:
   1. 更新状态（移动对象、刷怪、碰撞、计分）
   2. 重绘画面（repaint）
```
本项目用 `javax.swing.Timer` 驱动这个循环。理解了主循环，就理解了 90% 的游戏架构。

### 3.2 分层（类 MVC）
- **模型**：`Player/Zombie/Bullet`（对象 + 状态）
- **控制**：`GameController`（每帧怎么变）
- **视图**：`GamePanel`（怎么画 + 收输入）
分开后，控制器能脱离 GUI 直接测试——这就是为什么 `TestGameLogic` 能跑。

### 3.3 多态的实战价值
`GameObject` 接口让 `GamePanel` 用**同一种方式**对待所有对象：
```java
for (Zombie z : zombies) z.update();   // 统一调 update
for (Bullet b : bullets) b.draw(g);    // 统一调 draw
```
将来加"Boss/道具"，只要实现 `GameObject` 就能融进来——**这就是多态的意义**。

### 3.4 碰撞检测（圆形）
两个圆相撞 ⟺ 圆心距离 < 半径之和：
```java
double d = Math.hypot(b.x - z.x, b.y - z.y);
if (d < b.radius + z.radius) { /* 命中 */ }
```
比矩形碰撞简单，2D 游戏常用。

### 3.5 三角函数做瞄准与运动
- **瞄准**：鼠标位置 → 玩家朝向角 `angle = Math.atan2(dy, dx)`
- **射击/移动**：沿角度分解成速度 `vx = cos(angle)*speed, vy = sin(angle)*speed`
游戏离不开这点数学。

### 3.6 帧驱动 vs 时间驱动
本项目是**帧驱动**：刷怪按"每 90 帧"（不是"每 1.5 秒"）。好处是好测、好控；代价是帧率变了节奏会变（阶段⑥可改成时间驱动）。

### 3.7 集合安全删除
**遍历时删除元素会抛 `ConcurrentModificationException`**。解法：
```java
zombies.removeIf(Zombie::isDead);   // 遍历结束后批量删
```
而不是在 `for` 里直接 `list.remove(...)`。

### 3.8 回调（观察者模式雏形）
`GameController` 不知道窗口存在，只持有 `Runnable onGameOver`；血量归零时调一下。窗口注册这个回调来做结算。**解耦**：控制器不依赖 UI。

## 4. 踩坑 / 注意点

| 坑 | 现象 | 解法 |
|---|---|---|
| 遍历时删元素 | `ConcurrentModificationException` | `removeIf` 批量删 |
| 僵尸撞玩家不删 | 反复扣血、瞬间死 | 撞到就 `kill()` 该僵尸，下帧被 `removeIf` 清掉 |
| 结算回调多次触发 | 死一次弹好几个框 | `gameOverFired` 标志位保证只触发一次 |
| EDT 上弹模态框 | 阻塞 Timer 回调 | `SwingUtilities.invokeLater` 推迟到绘制后 |
| 存档抛异常 | 结算弹窗被吞 | `saveRecord` 套 try/catch |
| X 关窗 DISPOSE | 进程空挂 | `windowClosing` 回主菜单（或用 EXIT_ON_CLOSE） |
| 血量负数 | HUD 显示 -15 | `takeDamage` clamp 到 0 |
| Windows 测试 classpath | `/tmp/...` 在 MSYS 不转换 → ClassNotFound | 用 Windows 路径（`C:/...` 或项目内相对路径） |

## 8. 阶段④增强：局内"退出本局"按钮 + 统一结算

阶段④完成后又加了一个体验增强：**玩到一半也能退出，且和被打死一样结算存档**。

### 做了什么
- **`GamePanel` 右上角画一个"退出本局"按钮**（红色圆角矩形）：`paintComponent` 里绘制，`mousePressed` 里**先判断点击是否落在按钮区域**——是则触发退出回调、不发射子弹，否则正常开火。按钮横坐标按画布实际宽度动态贴右。
- **`GameWindow` 抽出 `settle()` 方法**：停循环 → 写战绩（try/catch 兜底）→ 弹"本局结算"框（再来一局 / 回主菜单）。**血量归零**和**退出本局**都走它。
- **`settled` 标志位**：防止"正好边退边死"在同一帧内重复触发结算、弹两次框。

### 三种离开方式的区别
| 方式 | 结算 / 存档 |
|---|---|
| 被打死（血量归零） | ✅ 结算 + 存档 |
| 点"退出本局"按钮 | ✅ 结算 + 存档 |
| 点窗口 X 关闭 | ❌ 不结算，直接回主菜单（快速退出） |

### 知识点
- **画布内"虚拟按钮"**：不用 Swing `JButton`，而是自绘 + 点击区域判定——和游戏画布融为一体，不被布局管理器干扰。
- **统一结算入口**：把多条退出路径收敛到一个 `settle()` 方法，避免逻辑重复、保证一致。
- **回调 + 标志位**：跨回调（Timer 触发的死亡 vs 鼠标触发的退出）去重，是事件驱动里常见的"只执行一次"套路。
