## 2. 核心知识点（复盘重点）⭐

### 3.1 JTable + DefaultTableModel（表格展示）
- `DefaultTableModel` 管列名和行数据；`addRow(Object[])` 加一行。
- **只读表格**：重写 `isCellEditable` 永远返回 `false`，用户就改不了单元格。
- `JTable` 放进 `JScrollPane` 才有滚动条和表头。

### 3.2 SQL JOIN（跨表取数据）⭐
排行榜要显示「**谁**打了多少分」，但 `game_record` 只有 `user_id`，没有昵称。用 JOIN 把 `user` 表的昵称拼过来：
```sql
SELECT g.*, u.nickname
FROM game_record g JOIN user u ON g.user_id = u.id
ORDER BY g.score DESC LIMIT ?
```
> JOIN 是关系型数据库的核心：把分散在多张表、靠外键关联的数据，按需拼成一张结果。

### 3.3 ResultSet 映射要「按列对齐」（本阶段最关键的坑）
`RecordDao` 原本有个共享的 `map(ResultSet)`。`topN` 加 JOIN 后结果集多了 `nickname` 列，而 `mine` 的结果集**没有**这列。如果在共享 `map` 里读 `nickname`，`mine` 会抛 `Column 'nickname' not found`。

**解法：拆分映射**
- `map(rs)`：只映射基础 6 列（`mine` 继续用，行为零变化）
- `mapWithUser(rs)`：先调 `map(rs)`，再补一行 `setNickname(rs.getString("nickname"))`（只有 `topN` 用）

> 复盘要点：**映射方法必须和它对应的 SQL 的列严格对齐**。SQL 多查了列，映射要单独处理，别让别的查询跟着踩雷。

### 3.4「仅显示用」字段（transient field）
`GameRecord.nickname` 不是数据库列，只是 JOIN 出来给界面显示用的。这种「模型里加一个非持久化字段」的做法很常见，注释里要标清楚「不是 DB 列」。

### 3.5 视图叠加（不 dispose 主菜单）
点「排行榜」时**不关主菜单**，排行榜作为独立窗叠加在上面；「返回」只 `dispose()` 排行榜窗，主菜单还在。比「关了再重建」省事、也更快。

## 4. 踩坑 / 注意点

| 坑 | 解法 |
|---|---|
| 共享 map 读 JOIN 列炸 mine | 拆分 `map` / `mapWithUser` |
| JOIN 语法记混 | `JOIN ... ON 外键关系`，ON 后面是两表的关联条件 |
| 昵称为空显示空白 | 空值回退到 `username`（和主菜单标题一致） |
| 表格能被双击编辑 | 重写 `isCellEditable` 返回 false |
| 命令行带 MySQL 密码 | ⚠️ 本阶段 DB 核对时把密码放命令行被安全扫描警告——本地验证低风险，但**别在会进仓库/日志的地方明文写密码** |
