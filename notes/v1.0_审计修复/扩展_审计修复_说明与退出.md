## 三、修复②：点 X 静默丢整局（[GameWindow.windowClosing](../../src/main/java/com/game/ui/GameWindow.java#L99)）

**问题**：点窗口 X 只 `panel.stop() + 回主菜单`，**不结算、不存档、不确认**；而暂停菜单的"退出游戏"按钮会存档+弹结算。同一个"离开本局"意图，**按钮存、X 不存**——误点 X 整局白打。

**修复**：X 也走"确认 → 存档 → 结算 → 回主菜单"，和按钮一致。这处改着改着踩了 3 个 Swing 的坑（对抗 agent 抓到的）：

### 知识点 ⭐

**① `DO_NOTHING_ON_CLOSE` + 手动 `dispose()` 才能做出"可取消的关闭"**
要让用户在"确认退出"对话框里点"否"时**窗口不关**，不能用默认的 `DISPOSE_ON_CLOSE`——它会在 `windowClosing` 返回后**自动销毁**窗口，你拦不住。改成 `DO_NOTHING_ON_CLOSE`，把"关不关"完全交给 `windowClosing` 里手动 `dispose()`：点"是"才 dispose，点"否"直接 return、窗口留着。
- 程序里别处主动调 `dispose()`（死亡结算、再来一局）**不受影响**——`DO_NOTHING` 只改"用户点 X"这条原生路径，代码 dispose 照常关窗。

**② 模态对话框【不会】自动暂停游戏 Timer ⚠️ 最关键**
原来以为"弹模态确认框 = 阻塞 EDT = Timer 自动停"。**错**。Swing 模态框跑的是**嵌套事件泵**（secondary loop），它**继续派发 EDT 队列里的事件**——包括 `javax.swing.Timer` 投递的 `tick()`。所以确认框弹出期间，**僵尸照常推进、玩家照常掉血/死亡**。
- 证据就在同文件：`openSettings()` 之所以**先 `panel.stop()` 再 `dialog.setVisible(true)`**，正是因为模态不会替你停 Timer。
- 所以 X 确认也必须**先 `panel.stop()`、取消时再 `panel.start()` 恢复**，否则玩家在犹豫"退不退"的几秒里可能就在背后被打死了。

**③ 模态泵里可能"闯入"已排队的回调 → 存档前复查 `settled`**
玩家死亡的当帧，`GameController.update` 会 `SwingUtilities.invokeLater(settle)` 把结算**排队**。如果死亡恰好发生在点 X 的同帧，这个排队的 `settle()` 会在**确认框的嵌套泵里被派发执行**（因为它继续派发 EDT 事件），中途把 `settled` 置 true、存了档、甚至 dispose 了窗口。
- 所以确认返回"是"后，**先 `if (settled) return;` 复查**，避免和闯入的 settle 重复存档/重开；
- 自己存档前 `settled = true`，挡住之后才跑的 settle（`settle()` 开头有 `if (settled) return`）。

**最终 windowClosing 流程**：`settled?→直接走` / 否则 `stop→确认→否:start 恢复` / 是:`settled?→return` / 否则 `settled=true→存档→结算框→dispose→主菜单`，和 `openSettings` 的"退出游戏"按钮完全对齐。

## 四、踩坑 / 收获

- **对抗验证真有用**：第一版 windowClosing 我自以为对了（编译过、逻辑自洽），对抗 agent 一查就抓到"模态不停 Timer"和"存档没加 settled 保护"两个真问题——这种"看起来对、实际有坑"的并发/事件循环交互，靠自己 review 容易漏，独立眼光值回票价。
- **Swing 的"暂停"要手动做**：只要游戏逻辑挂在 `javax.swing.Timer`（EDT）上，任何"想暂停一下问用户"的弹窗都得自己先 `stop`，别指望模态框替你停。
- **可取消的窗口关闭 = `DO_NOTHING_ON_CLOSE` + 手动 dispose**，这是个常用套路（确认退出/保存提示都靠它）。
- **文案即代码**：玩家可见的文字（说明、按钮名、错误提示）和实现脱节也是 bug，而且往往是 demo 时最先被发现的。
