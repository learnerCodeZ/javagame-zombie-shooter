## 一、需求与两个关键决策

用户要求：**昵称保留，注册不用用户名、改用 11 位电话**。

### 决策 1：手机号校验放宽为「11 位数字」`\d{11}`（而非严格手机号 `1[3-9]\d{9}`）
最初我按正规手机号写了 `1[3-9]\d{9}`。但用户随后指定 **admin 手机号 = `00000000000`、z 手机号 = `11111111111`**（便于记忆的测试值），这两个都不符合严格号段。
- 若保留严格手机号校验 `1[3-9]\d{9}`：admin(`00000000000`)、z(`11111111111`) 这类记忆值在注册/登录/忘记密码任一处都会被拒——**种子数据和校验规则自相矛盾**。
- 结论：校验必须与种子数据一致。用户要的本质就是「**11 位数字**」，所以放宽到 `phone.matches("\\d{11}")`。这也更贴合最初「11 位电话」的字面要求。

> **教训**：**输入校验规则要和你写进库的种子数据自洽**。种子里的值如果连自己的校验都过不了，就是个埋着的雷。

### 决策 2：admin=`00000000000`、z=`11111111111`，其余种子不变
- `schema.sql` 种子：admin→`00000000000`、player1→`13800000001`、player2→`13800000002`（admin 单独记忆值）。
- 历史注册用户 `z`（非种子、id=5）在**线上库**直接 `UPDATE` 成 `11111111111`（z 不进 schema，只是改活库）。

## 三、数据库迁移（[`migrate_v1.3_phone.sql`](../../sql/migrate_v1.3_phone.sql)）

把 `user.username` 列改名为 `phone`。**幂等**（information_schema 判断旧列是否还在，已迁移则全跳过）。三个步骤都按「第 1 次才执行」护栏：

1. **先把旧列的值改写成 11 位**（`CONCAT('138', LPAD(id-1, 8, '0'))`，admin 再覆盖成 `00000000000`）——**必须先改数据、再缩列宽**，否则 `VARCHAR(50)→VARCHAR(11)` 遇到超长旧值会在严格模式下报错。
2. **重命名列** `CHANGE COLUMN username phone VARCHAR(11) NOT NULL`（列上的唯一索引随列保留）。
3. **唯一索引改名** `uk_username → uk_phone`。

```
-- 幂等开关：旧列还在吗？
SET @has_username = (SELECT COUNT(*) FROM information_schema.columns
  WHERE ... column_name='username');
SET @s1 = IF(@has_username=1, "UPDATE user SET username=CONCAT(...)", 'DO 0');  -- 先改数据
... admin 覆盖成 00000000000 ...
SET @s2 = IF(@has_username=1, "ALTER TABLE user CHANGE username phone VARCHAR(11)...", 'DO 0');  -- 再改名
```

> **`schema.sql`（全新安装）** 已直接建成 `phone VARCHAR(11)` + `uk_phone`，种子 admin=`00000000000`。
> **线上库**：跑了一次 migrate_v1.3（admin→00000000000），再 `UPDATE z→11111111111`；迁移连跑第 2 次无报错（幂等验证通过）。
> **历史迁移顺序**：migrate_v1.1 里有 `WHERE username='admin'`（旧列名），v1.3 之后**勿再重跑 v1.1**；全新机直接用 schema.sql。

## 五、踩坑 / 收获

- **标识符改名是横切关注点 ⭐**：一个字段名牵动 model→DAO→UI→test→schema→迁移→6 份文档。**先用 `grep` 把所有引用找全**再动手，否则漏一处就编译失败或文档说谎。本次用 `getUsername|setUsername|findByName|用户名|username` 多个模式扫了两轮。
- **校验规则要和种子数据自洽**：严格手机号正则会拒掉 admin(`00000000000`)、z(`11111111111`)——种子能过校验是底线。最终放宽到 `\d{11}`，既满足「11 位电话」、又让记忆值合法。
- **幂等的「列改名」迁移套路**：`information_schema` 判断旧列在不在做开关 + **先改数据再缩列宽**（避免超长报错）+ 数据改写只在首次迁移跑（用同一个 `@has_username` 护栏，重跑不覆盖用户改过的手机号）。
- **历史复盘笔记保持原样**：02/03/05/扩展_用户管理/审计修复 等旧笔记里的「用户名」描述不改——它们是**当时的记录**；新改动写**新笔记**（本篇），并在文档里指出「以本篇为准」。只有**面向未来的答辩准备 00 总纲**跟着改了（那是你要照着背的）。
