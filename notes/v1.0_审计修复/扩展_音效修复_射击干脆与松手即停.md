## 一、问题
- ① 连按射击时声音"乱"（不干脆、有爆音）。
- ② 松开鼠标后，声音还在响一会儿。

## 二、根因诊断 ⭐（都在 SourceDataLine 的缓冲）
当前模型：单条 `SourceDataLine` + 队列 + 守护消费线程；`shoot()` 有 40ms 防抖 + 队列上限 2。

**根因 A（松开还在响，主凶）**：`ensureLine()` 用 `l.open(FORMAT)` 打开线路，**没指定缓冲区大小 → 用 JVM 默认（通常几百 ms）**。关键事实：**`line.write()` 只在缓冲"满"时才阻塞**。于是消费线程把队列里的音效**以远快于实时的速度**全灌进线路缓冲，线路再按实时慢慢播 → 音频被"囤积/延迟"。连射时线路缓冲攒了一串射击音，**松手后这些缓冲里的声音继续往外吐 → "还在响"**。
> 注意：**队列上限 2 只管队列，管不了线路缓冲**——队列被抽干进了缓冲，缓冲照样囤。

**根因 B（连按乱响）**：① 缓冲把多个音"抹"在一起延迟；② 每个音**只有开头淡入（attack）、没有结尾淡出（release）**——末尾包络还在 ~37% 振幅，相邻音衔接处振幅突变 → 咔哒/爆音，连射一堆爆音 = "乱"。

## 三、修复（两处，低风险）
1. **线路开小缓冲**：`l.open(FORMAT)` → `l.open(FORMAT, LINE_BUFFER_BYTES)`，`LINE_BUFFER_BYTES ≈ 4410`（= 44100Hz × 2 字节 × 50ms）。让 `write()` 在缓冲接近满时阻塞 → 消费线程被**实时节流**，队列无法快于播放被抽干 → **音频几乎不囤积、松手即停**。50ms 延迟人耳察觉不到，又足够大避免欠载爆音。
2. **结尾淡出（release）**：`buildTone` / `buildSweep` 末尾约 5ms（`RELEASE_SEC`）振幅线性降到 0，与开头 attack 对称 → 相邻音衔接处归零、无突变 → **连射不咔哒**。

> 保留原 40ms 防抖 + 队列上限 2（仍防连射堆爆）。小缓冲不会让"有时候没声音"复发——那是当年开太多线路导致的，现在是单线路复用，与小缓冲无关。

## 四、为什么没加 flush-on-shoot（权衡后放弃）
考虑过"每次 `shoot()` 先 `line.flush()` 丢弃缓冲里未播的音频再写新的"——最跟手、完全不叠。但代价不划算：
- **切掉命中/受伤音**：线路共用，flush 一刀下去，缓冲里排队的命中/受伤音也被丢 → 战斗中"打中没有反馈"、挨打了没听清。
- **重新引入爆音**：flush 在波形任意位置硬切，切断点振幅突变 → "咔"一声，把刚加的淡出白做一部分。
- **连射变"嗒嗒嗒"**：每个 pew 只剩起声、身体被下一个 flush 切掉。

→ 用户实机试了"小缓冲 + 淡出"版觉得"还可以"，flush 没必要加。

## 五、踩坑 / 收获（Java Sound 知识点）
- **`SourceDataLine.write()` 只在缓冲满时阻塞**——最反直觉的一点。默认大缓冲会让"生产者（队列）"远快于"消费者（扬声器）"地往缓冲灌数据，造成延迟/囤积。**要低延迟，必须主动开小缓冲 `open(format, smallBytes)`**。
- **队列限流 ≠ 线路限流**：队列上限挡不住线路缓冲囤积，因为消费者会把队列抽干进缓冲。
- **音效要首尾都归零**：只有 attack（淡入）不够，必须有 release（淡出），否则相邻音拼接处咔哒。起声 + 释音对称是合成短音的基本功。
- **诊断要找准"延迟发生在哪一层"**：这里延迟在**线路缓冲**、不在队列——找准了才能对症（开小缓冲）。
