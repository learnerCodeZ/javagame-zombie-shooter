## 二、修复③：deleteUser 三表删除包事务（[UserDao.deleteUser](../../src/main/java/com/game/dao/UserDao.java#L152)）
**问题**：删用户要清 `game_record` / `password_reset_request` / `user` 三张表，原代码走 autocommit、每条 DELETE 即时提交。若第三条（user）失败，前两张已提交 → **账号还在、战绩和重置申请已被永久清空**（孤儿数据）。

**修法**：照同模块 [`ResetRequestDao.approve`](../../src/main/java/com/game/dao/ResetRequestDao.java#L120) 的事务写法——`setAutoCommit(false)` → SELECT role + 三条 DELETE → `commit()`，catch 里 `rollback()`，finally 还原 autoCommit + close。

**对抗验证抓到的 refinement ⭐**：第一版「无条件 commit」仍有缝——若第三条 DELETE 因并发（role 被改成 admin）影响 0 行，仍会 commit 前两步、制造孤儿。改成 **`if (deleted) commit(); else rollback();`**：只有真正删掉才提交，否则连子表删除一起回滚。这才是事务真正要保证的「要么全删、要么一点不动」。

> **知识点**：事务原子性 = all-or-nothing；`setAutoCommit(false)` 手动控制提交点；finally 必须还原 autoCommit + close 防连接泄漏；**commit 的条件要想清楚**——不是"执行完就 commit"，而是"达到预期效果才 commit"。

## 三、修复④：僵尸贴墙凭空生成扣血（[GameController.spawnZombie](../../src/main/java/com/game/game/GameController.java#L199)）
**问题**：僵尸在画布四边生成，玩家贴墙时（中心距边 26px），若该边新生成僵尸的随机坐标落在玩家附近约 35px 内，**第一帧就圆心距 < 碰撞阈值、立即扣血 + 僵尸消失**——玩家看不到、躲不开。

**修法**：把"选边 + 选坐标"包进最多 10 次的重试循环，只在生成点离玩家 ≥ `safeDist`（`玩家半径 + 僵尸半径 + 40` 缓冲 ≈ 80~92px）时才采用。

> **已知小注记**：`safeDist` 里的僵尸半径 `brute ? 26 : 18` 是硬编码，重复了 Zombie 类的规则——目前值一致、保护有效，但日后改 Zombie 半径要记得同步这行（理想做法是抽成 `Zombie.radiusFor(type)` 统一来源）。属维护性提示，非现行 bug。

> **知识点**：「生成即碰撞」是 2D 射击的经典不公平；生成时做距离剔除/安全区是标准对策。

## 四、修复⑤：注册长度校验（[RegisterFrame.doRegister](../../src/main/java/com/game/ui/RegisterFrame.java#L104)）
**问题**：schema 里 `username`/`nickname` 是 `VARCHAR(50)`，注册界面只校验非空和密码 ≥ 6，不限制长度。输入 > 50 字符时 MySQL 严格模式抛 `SQLException`，被 catch 吞成 `return false` → 界面弹"注册失败（用户名可能已存在）"——**实际是超长，错误信息完全误导**。

**修法**：在 DAO 调用前加 `username.length() > 50`、`nickname.length() > 50` 校验，弹准确的"不能超过 50 个字符"。

> **知识点**：输入校验要在「靠近入口」做（界面层），别让非法值一路捅到 DB 再被约束拒、报出无关错误；**校验上限要和 schema 一致**（VARCHAR(50) ↔ ≤ 50）。

## 五、修复⑥：暂停菜单"退出游戏"→"回主菜单"（[GameWindow.openSettings](../../src/main/java/com/game/ui/GameWindow.java#L261)）
**问题**：该按钮实际是"存档 + 回主菜单"，却叫"退出游戏"——而死亡结算里同一去向叫"回主菜单"。文案不一致，"退出游戏"还让人以为会关掉整个程序。

**修法**：统一文案为"回主菜单"（按钮 + 确认框"确定回主菜单?"），与死亡结算一致。

**颜色决策**：对抗验证指出——若把红改成绿，会和"再来一局"撞色失去视觉区分，也脱离项目"退出/离开 = 红"的约定（"退出登录"也是红）。所以**保留红色、只改文案**：红色"回主菜单"既保持与"再来一局"的区分、又符合"离开本局"的语义，标签清晰不会误导成"关程序"。

> **知识点**：改文案时，颜色/语义别一并乱动；**标签准确 + 视觉区分**要兼顾，别修了一个引入另一个。

## 六、踩坑 / 收获
- **「包了事务」≠「原子性达标」**：第一版事务仍 commit 了 0 行删除的孤儿路径——对抗验证才抓到。事务要保证"要么全成、要么全废"，**commit 的条件要想清楚**（这次是 `if (deleted)`，不是执行完就提交）。
- **对抗验证又立功**：4 处修复它抓到 commit 漏洞、颜色撞色、半径硬编码、笔记过期 4 条，都是自己 review 容易漏的边角。
- **生成即碰撞**：2D 游戏要在生成点做安全距离剔除，否则贴墙玩家会被"凭空"伤害，很不公平。
- **校验靠前 + 与 schema 对齐**：别让脏数据捅到 DB 报无关错误。
